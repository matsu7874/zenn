# デファクトスタンダードなcrate群

Rustのエコシステムも成熟しており下記の領域についてはデファクトスタンダードなライブラリ（Rustではcrateと呼ぶ）が存在する。いずれも数億ダウンロード級のcrate群です。
これらの領域で他のライブラリ選定を行う場合、作ろうとしているシステムの特性に沿っているか、なぜデファクトスタンダードのライブラリ選定ではないのか理由を聞いてみるべきです。

| 領域 | デファクト | 補足 |
|---|---|---|
| シリアライズ | [serde](https://crates.io/crates/serde) (+ [serde_json](https://crates.io/crates/serde_json)) | 型とJSON等データフォーマットの相互変換用crate |
| エラー処理(アプリ) | [anyhow](https://crates.io/crates/anyhow) | アプリ向けのエラーを伝搬するためのcrate |
| エラー処理(ライブラリ) | [thiserror](https://crates.io/crates/thiserror) | ライブラリ向けのエラー定義crate。anyhowと併用が定石 |
| CLI引数パース | [clap](https://crates.io/crates/clap) (derive) | 競合であった[structopt](https://crates.io/crates/structopt)は clap v3 に統合済み |
| HTTPクライアント | [reqwest](https://crates.io/crates/reqwest) | `features = ["rustls-tls"]` 指定でOpenSSL依存を回避。同期で十分なら依存最小の [ureq](https://crates.io/crates/ureq) という選択肢もある |
| 非同期ランタイム | [tokio](https://crates.io/crates/tokio) | 対抗だった[async-std](https://crates.io/crates/async-std)は開発停止。現状一択 |
| 正規表現 | [regex](https://crates.io/crates/regex) | カバーされていない拡張表現がなければ、これが標準 |
| データ並列 | [rayon](https://crates.io/crates/rayon) | `par_iter()` に変えるだけで並列化 |

※2026年6月時点

# ロギング：順番に見るか絞り込んで見るか
前節では一つの目的に一つのデファクトスタンダードcrateが対応していたが、当然そうではない領域も当然存在します。ロギングはその中でもシンプルであるため、最初に紹介します。

ロギングについては、その目的に応じてcrateを使い分ける必要があります。ログをどのように見ていきたいかによって、デファクトスタンダードなライブラリが2群(log系、[tracing](https://crates.io/crates/tracing))あります。
プログラムの動作を出力するという観点ではprintln!デバッグ系も合わせてそれぞれ特徴を見てみましょう。

## println! デバッグ

自分でプログラミングをしていた頃にはとりあえず入れて観察するかために使っていたプリントデバッグですが、デバッグが必要になってからAIにこれを入力させて問題を分析するのは効率が悪いです。
普段からログを吐いておいてLLMが見れるようにしておくべきでしょう。

```rust
let user_id = 1;

println!("user_id = {user_id}"); // => user_id = 1
dbg!(user_id);                  // => [src/main.rs:4:1] user_id = 1
```

## log系

[`log`](https://crates.io/crates/log) crateは「ログを出す側のAPI」と「ログを処理する実装」を分離しています。ライブラリは `log::info!` だけ書き、アプリケーション側が [`env_logger`](https://crates.io/crates/env_logger) などのロガー実装を選ぶ仕組みになっています。
シンプルに順番に読めば理解できるログを出すなら、未だにlogクレートとenv_loggerなどの組み合わせが一般的です。

ログフォーマットは編集できますが、標準的には下記のようにタイムスタンプとメッセージが表示されます。
```
[2026-06-15T08:01:46Z INFO  log_sample] アプリケーション起動
[2026-06-15T08:01:46Z INFO  log_sample] 注文を確定しました: order_id=1001, qty=2
[2026-06-15T08:01:46Z WARN  log_sample] 数量が 0 の注文を受け取りました: order_id=1002
[2026-06-15T08:01:46Z ERROR log_sample] 注文処理に失敗: order_id=1003, reason=在庫不足: 要求=99, 在庫=10
[2026-06-15T08:01:46Z INFO  log_sample] 全注文の処理が完了
```

これで十分ならlogとenv_loggerを使うのが第1候補になります。

## tracing

複数のワーカーで大量のファイルを処理させたり、複数のリクエストに対応するWebバックエンドなどは [tracing](https://crates.io/crates/tracing) を使うのが一般的です。
このようなタイプのアプリケーションでは複数の処理が同時並行で進むため、タイムスタンプ順にログを見るのではなく、あとから処理単位で絞り込んでログを分析することが多く、それぞれのログがどの処理単位に属するのかを提供したcrateを使う動機があります。

tracingではどのリクエストに対応するログであるかを記録することが簡単です。
例として3つのリクエストに対応するWebバックエンドのログを示します。
