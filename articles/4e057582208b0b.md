AIの発展によって自分が詳しくない領域についても自分用のツールを作ることができるようになりました。
その際に一番困ることはライブラリ選定の良し悪しが分からないことです。

特にプログラミング言語としてRustを採用する場合、標準ライブラリが比較的薄いこともあり、毎回外部のライブラリを導入しているように見え、不安を感じる方もいるでしょう。
この連載では主に普段Rustでプログラミングをしていない人向けに、AIが行ったライブラリ選定をチェックする基準を示します。

# デファクトスタンダードなcrate群

Rustのエコシステムも成熟しており下記の領域についてはデファクトスタンダードなライブラリ（Rustではcrateと呼ぶ）が存在する。いずれもN億ダウンロード級のcrate群です。
これらの領域で他のライブラリ選定を行う場合、作ろうとしているシステムの特性に沿っているか、なぜデファクトスタンダードのライブラリ選定ではないのか理由を聞いてみるべきです。

| 領域 | デファクト | 補足 |
|---|---|---|
| シリアライズ | serde (+ serde_json) |  |
| エラー処理(アプリ) | anyhow | アプリ向けのエラーを伝搬するためのcrate |
| エラー処理(ライブラリ) | thiserror | ライブラリ向けのエラー定義crate。anyhowと併用が定石 |
| CLI引数パース | clap (derive) | 旧世代のstructoptは clap v3 に統合済み |
| HTTPクライアント | reqwest | `features = ["rustls-tls"]` 指定でOpenSSL依存を回避(軸8)。同期で十分なら依存最小の ureq という選択肢もある(軸7) |
| 非同期ランタイム | tokio | 対抗だったasync-stdは開発停止。一択 |
| 正規表現 | regex | カバーされていない拡張表現がなければ、これが標準 |
| データ並列 | rayon | `par_iter()` に変えるだけで並列化 |


# ロギングは順番に見るか絞り込んでみるかで考える

ロギングについては、ログをどのように見ていきたいかによってデファクトスタンダードなライブラリが2群(log系、tracing)あります。
プログラムの動作を出力するという観点ではprintln!デバッグ系も合わせてそれぞれ特徴を見てみましょう。

## println! デバッグ

自分でプログラミングをしていた頃にはとりあえず入れて観察するかために使っていたプリントデバッグですが、デバッグが必要になってからAIにこれを入力させて問題を分析するのは効率が悪いです。
普段からログを吐いておいてLLMが見れるようにしておくべきでしょう。

```rust
let user_id = 1;

println!("user_id = {user_id}"); // => user_id = 1
dbg!(user_id);                  // => [src/main.rs:4:1] user_id = 1
```

## log系

`log` crateは「ログを出す側のAPI」と「ログを処理する実装」を分離しています。ライブラリは `log::info!` だけ書き、アプリケーション側が `env_logger` などのロガー実装を選ぶ仕組みになっています。
シンプルに順番に読めば理解できるログを出すなら、未だにlogクレートとenv_loggerなどの組み合わせが一般的です。

ログフォーマットは編集できますが、標準的には下記のようにタイムスタンプとメッセージが表示されます。
```
[2026-06-15T08:01:46Z INFO  log_sample] アプリケーション起動
[2026-06-15T08:01:46Z INFO  log_sample] 注文を確定しました: order_id=1001, qty=2
[2026-06-15T08:01:46Z WARN  log_sample] 数量が 0 の注文を受け取りました: order_id=1002
[2026-06-15T08:01:46Z ERROR log_sample] 注文処理に失敗: order_id=1003, reason=在庫不足: 要求=99, 在庫=10
[2026-06-15T08:01:46Z INFO  log_sample] 全注文の処理が完了
```

## tracing

複数のワーカーで大量のファイルを処理させたり、複数のリクエストに対応するWebサーバーなどはtracingを使うのが一般的です。
このようなタイプのアプリケーションでは複数の処理が同時並行で進むため、タイムスタンプ順にログを見るのではなく、あとから処理単位で絞り込んでログを分析することが一般的です。
したがってそれぞれのログがどの処理単位に属するのかを提供したcrateを使う動機があります。

tracingではどのリクエストに対応するログであるかを記録することが簡単です。
例として3つのリクエストに対応するWebサーバーのログを示します。
```
2026-06-15T08:05:08.644411Z  INFO tracing_sample: アプリケーション起動
2026-06-15T08:05:08.644904Z  INFO process_order{order_id=1003 qty=99 stock=10}: tracing_sample: 受付
2026-06-15T08:05:08.644910Z  INFO process_order{order_id=1001 qty=2 stock=10}: tracing_sample: 受付
2026-06-15T08:05:08.644919Z  INFO process_order{order_id=1002 qty=0 stock=10}: tracing_sample: 受付
2026-06-15T08:05:08.675738Z  INFO process_order{order_id=1001 qty=2 stock=10}: tracing_sample: 在庫を確認中
2026-06-15T08:05:08.679629Z  WARN process_order{order_id=1002 qty=0 stock=10}: tracing_sample: 数量が 0 の注文を受け取りました
2026-06-15T08:05:08.679730Z  INFO process_order{order_id=1002 qty=0 stock=10}: tracing_sample: return=Ok(())
2026-06-15T08:05:08.679943Z  INFO process_order{order_id=1003 qty=99 stock=10}: tracing_sample: 在庫を確認中
2026-06-15T08:05:08.707907Z  INFO process_order{order_id=1001 qty=2 stock=10}: tracing_sample: 注文を確定しました
2026-06-15T08:05:08.707962Z  INFO process_order{order_id=1001 qty=2 stock=10}: tracing_sample: return=Ok(())
```
