SWEの松本([@matsu7874](https://twitter.com/matsu7874))です。連載最終回は GUI crate を扱います。これまで（[第1回](https://zenn.dev/estie/articles/4e057582208b0b)デファクトcrate群、[第2回](https://zenn.dev/estie/articles/79aacb4bc57e69)Web/DB、[第3回](https://zenn.dev/estie/articles/2c987dd1a96865)時刻）と違い、GUIは正解の選択肢が用途で大きく変わる、もっとも割れる領域です。

結論から書くと、GUIのcrate選定は次の2点でほぼ決まります。

1. ビルドが通る環境を作れるか（Pure Rust か、外部のCライブラリや WebView が要るか）
2. AIの修正ループが回るか（GUIは進化が速く一発では当たらない。外したときに修正できるか）

実際に最新版へ書かせて検証すると、即時モードで「AIと相性が良い」とされる egui ですら、一発書きは 0/5 で全滅でした。理由は後述しますが、GUIを一発書きの精度で選んではいけないということです。

# 4つの観点（GUI版）

第2回から使っている4観点を再掲します。見るのは、AIに書かせたときに実装ループが回るかです。

| 観点 | チェック内容 | NG例 |
|---|---|---|
| AI親和性 | AIが現行APIでコードを書け、エラーを渡せば修正できる | 旧APIを出し、しかも原因が環境で修正できない |
| 立ち上げと反復速度 | READMEからシンプルに動き、build と `cargo check` の待ちが短い | C依存・WebView前提で詰まる／依存が重くビルドが長い |
| エコシステム適合 | tokio/serde等と自然に繋がる | 独自世界観が強すぎる |
| 撤退可能性 | adapterで隔離でき、メンテが止まっても捨てられる | UI構造がアプリ全体に広がる |

GUIで特に重要なのは、立ち上げと反復速度（C依存や外部依存でビルドが通るか）と AI親和性（学習データがそのバージョンに追いついているか）の2つです。加えてGUIならではの観点として、ビルドが通った後にAIや自動テストで動作の検証ループを閉じられるかも見ます（後述）。

# GUIは設計思想で4つに分ける

GUIライブラリは数が多いので、設計思想の代表でまず4つに整理します。これにC依存の教材として gtk4 を加えます。

| パラダイム | crate | 描き方 |
|---|---|---|
| 即時モード | [egui](https://crates.io/crates/egui)（[eframe](https://crates.io/crates/eframe)） | 毎フレームUIを記述。状態管理が単純 |
| Elm Architecture | [iced](https://crates.io/crates/iced) | state / Message / update / view に分ける |
| 宣言的DSL | [slint](https://crates.io/crates/slint) | 独自の `.slint` 言語でUIを書きコンパイル |
| Web技術 | [tauri](https://crates.io/crates/tauri) | HTML/CSS/JS + Rust。OSのWebViewを使う |
| ネイティブ(C) | [gtk4](https://crates.io/crates/gtk4) | GTK4へのバインディング |

※バージョンは eframe/egui 0.34 / iced 0.14 / slint 1.16 / tauri 2.11 / gtk4 0.11（2026年6月時点）

ここでも見るのは、AIに書かせる前提での回りやすさです。同じカウンタ（+1 / -1 ボタンと現在値）を各crateで一発書きさせ、`cargo build` の通過で評価しました。

## egui（eframe）の即時モード

egui は「毎フレーム、いまのUIを全部記述する」即時モードです。状態のツリーを管理しなくてよく、AIとの相性が良いと一般に言われます。

```rust
// eframe 0.34（2026年時点）の正しい書き方
use eframe::egui;

struct CounterApp { count: i64 }

impl eframe::App for CounterApp {
    fn ui(&mut self, ui: &mut egui::Ui, _frame: &mut eframe::Frame) {
        ui.label(format!("Count: {}", self.count));
        if ui.button("+1").clicked() { self.count += 1; }
        if ui.button("-1").clicked() { self.count -= 1; }
    }
}

fn main() -> eframe::Result<()> {
    eframe::run_native("Counter", eframe::NativeOptions::default(),
        Box::new(|_cc| Ok(Box::new(CounterApp { count: 0 }))))
}
```

ところが、このコードには注意点があります。多くの学習データにある `fn update(&mut self, ctx, frame)` 形式は、eframe 0.34 ではもうコンパイルが通りません（必須メソッドが `ui` に刷新されました）。これが後述の「0/5」の原因です。

## iced の Elm Architecture

iced は state / Message / update / view に分ける、Elm由来のアーキテクチャです。状態遷移が型（`Message` enum）で表現され、規律ある設計ができます。

```rust
// iced 0.14 の正しい書き方（application の第1引数が状態初期化関数）
fn main() -> iced::Result {
    iced::application(Counter::default, Counter::update, Counter::view)
        .title("Counter")
        .run()
}
```

iced にも同じ種類の注意点があります。0.14 で `application` の第1引数が「タイトル文字列」から「状態初期化関数」に変わりました。`iced::application("Counter", ...)` という旧来の形は通りません。

## slint の宣言的DSL

slint は独自の `.slint` 言語でUIを宣言的に書き、ビルド時にネイティブコードへコンパイルします。`slint!` マクロを使えば Rust ファイル内に直接書けます。

```rust
slint::slint! {
    import { Button, VerticalBox } from "std-widgets.slint";
    export component MainWindow inherits Window {
        // ... プロパティとコールバック
    }
}
```

独自DSLなので学習データは相対的に薄いはずですが、後述のとおり一発書きは健闘しました。なお slint はライセンスが3択（Royalty-Free / GPLv3 / 商用）で、配布形態によっては自分のコードのライセンスに影響する点は選定上の注意です。

## tauri は Web技術と OS WebView を組み合わせる

tauri はフロントエンドをHTML/CSS/JSで書き、RustバックエンドとOSのWebView（macOSなら WKWebView）を組み合わせます。フロントの学習データは膨大で、バイナリにブラウザエンジンを同梱しないため小さくなります（Electronとの大きな違い）。

```rust
#[tauri::command]
fn increment(value: i64) -> i64 { value + 1 }

fn main() {
    tauri::Builder::default()
        .invoke_handler(tauri::generate_handler![increment])
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
```

ただし tauri は最小構成でも段取りが多い（後述）。

# 4つの観点で評価する

最新版で実際に書かせて採点した結果を反映した評価です。

| 観点 | egui | iced | slint | tauri | （gtk4） |
|---|---|---|---|---|---|
| AI親和性 | 一発0/5（新API未追従）。ヒントで1ラウンド復旧 | 一発0/5（新API未追従）。ヒントで1ラウンド復旧 | 4/5。DSLエラーも1ラウンドで復旧 | Web技術は学習厚く、Rust側は修正できる | コードは機械的だが、原因がC依存で環境ループ |
| 立ち上げと反復速度 | Pure Rustで素で通る・eframe雛形で即（208依存） | Pure Rustで素で通る・雛形あり（191依存） | Pure Rustで通る・`slint!`で完結（335依存でやや重い） | CLI/設定/アイコン/HTML/WebView(Xcode)一式が要る（272依存） | pkg-config+GTK必須。素では一歩も進めない |
| エコシステム適合 | 素直 | tokio等と素直 | DSL世界観 | Web資産/tokio | GTK世界観 |
| 撤退可能性 | 隔離しやすい | MVU構造が広がる | DSL層が挟まる | フロント/バック分離で剥がしやすい | GTK資産次第 |
| 検証ループの閉じやすさ | a11yツリーで駆動・スナップショット可 | `update`純粋関数が強い＋`simulator` | 専用ハーネスでドラッグも一発・完全headless | DOMでWeb資産。ただしWebView越しE2EはmacOS非対応 | ビルド不可で検証以前 |

※一発書きの精度を実測したのは、単一バイナリで完結する egui / iced / slint です。tauri（Rust＋Web＋設定の複合）と gtk4（環境構築で止まる）は一発書きの粒度に合わないため、ビルド、依存、環境の観点で評価しています。

※最終行「検証ループの閉じやすさ」は、4観点に加えたGUI固有の軸です（ビルドが通った後、自動テストやAIで動作を確かめられるか）。実測は後述の「ビルドが通った後の検証ループを誰が閉じるか」で扱います。

## 実測：一発書きの成績

Claude Opus 4.8 に、ドキュメント参照を禁じた「一発書き」で egui / iced / slint 各5試行、`cargo build` の通過で判定しました。

学習データの所在から、当初は「egui（即時モードで単純）が一発書きに強い」と予想していました。予想は外れました。

- **egui: 0/5**。5試行すべてが、eframe 0.34 で廃された旧 `fn update` 形式で書き、`E0046 missing ui` でコンパイルエラー
- **iced: 0/5**。5試行すべてが、0.14 で変わった `application` の第1引数を旧来のタイトル文字列で渡し、`E0277 &str: BootFn` でコンパイルエラー
- **slint: 4/5**。独自DSLながら健闘。失敗1件も `std-widgets` の import 漏れ

GUIは進化が速く、最新版でAPIが変わると、学習データが追いつかず一発では当たりません。「即時モードだからAIと相性が良い」という一般論は、バージョンが動くと崩れます。

ただし、失敗は全てコンパイルエラーでした（第3回の jiff のような silent failure はゼロ）。失敗11件すべて、コンパイラの出力（`help:` 行を含む）だけを返すと1ラウンドで build が通っています。egui の `help: implement the missing item: fn ui(...)` が決定打になりました。

一発書きの精度だけ見れば egui / iced は最低ですが、修正ループは1ラウンドで収束しました。第2回・第3回と同じく、一発書きの精度より復旧ループの回りやすさで評価すべきだということが、GUIでも再現したわけです。

# ビルドと配布がGUI選定を支配する

GUI選定で最も差が付くのがこの観点です。実測した依存数、クリーンビルド時間、バイナリサイズを示します。

| crate | ユニーク依存数 | release クリーンビルド | release バイナリ | 外部依存 |
|---|---|---|---|---|
| iced | 191 | 20.4s | 10.2 MB | Pure Rust |
| egui (eframe) | 208 | 22.2s | 11.5 MB | Pure Rust |
| tauri | 272 | 28.5s | **8.1 MB** | OS WebView・要Xcode |
| slint | 335 | 32.7s | **8.4 MB** | Pure Rust |

時刻crate（第3回は依存2〜15）と比べ、GUIは 191〜335依存 と桁違いに重く、クリーンビルドは20〜33秒、バイナリは8〜11.5MBです。バイナリは tauri / slint が小さめです。tauri はOSのWebViewを使うためブラウザを同梱せず、Web技術系でも肥大しません。

## gtk4 のC依存の壁

第2回で diesel が `ld: library 'pq' not found` でリンク失敗したのと同じ構図が、GUIではより大規模に起きます。pkg-config と GTK4 が入っていない素のmacOSで gtk4 をビルドすると、次のようになります。

```
warning: glib-sys@0.22.6: Could not run `pkg-config --libs --cflags glib-2.0 ...`
  The pkg-config command could not be found.
error: failed to run custom build command for `glib-sys`
（gobject-sys / gio-sys / gdk-pixbuf-sys / pango-sys / cairo-sys-rs /
  graphene-sys / gdk4-sys / gsk4-sys / gtk4-sys も同様に失敗）
```

10個の `*-sys` crateが軒並み pkg-config と GTK 一式を要求して全滅します。AIエージェントにとって、これはコードの修正では解決せず、環境構築のループに迷い込むパターンです。一方、egui / iced / slint は Pure Rust なので、`cargo build` が通る環境さえあれば環境構築ごとAIに任せられます。

## tauri は設定とアセットの段取りで中間に位置する

tauri は依存こそ Pure Rust 寄りです。しかし最小のカウンタをビルドするだけでも、アイコン画像、`tauri.conf.json`、`build.rs`、フロントエンドのHTML、OS WebView(Xcode) が必要です。実際、本検証でも「`dist` が無い」「`icon.png` が無い」で2回足踏みしてからビルドが通りました。`cargo tauri` のような専用CLIで雛形を生成するのが前提になります。初速の観点では、Pure Rust の egui/iced/slint に一歩譲ります。

# ちょっとしたTodoで見る「整いやすさ」

カウンタのような最小例では差が出ませんが、実用アプリでは、整ったものがどれだけ手早く作れるかも見たいところです。同じ仕様のTodo（テキスト入力、追加、完了チェック、削除、残数表示）を egui / iced / slint で書き、デフォルトの見た目を比べました。

| | egui | iced | slint |
|---|---|---|---|
| デフォルトテーマ | 素朴（ダーク） | 整ったライト | モダン（ネイティブ風） |
| 日本語表示 | **豆腐（□）になる** | デフォルトでOK | デフォルトでOK |
| 標準ウィジェット | 入力/チェック/ボタン一通り | 一通り＋装飾済み | 一通り＋装飾済み |

![egui Todo（日本語が豆腐）](/images/gui-todo-egui.png)
*egui: 既定フォントにCJKグリフが無く、日本語が □ になる*

![iced Todo](/images/gui-todo-iced.png)
*iced: 既定で日本語が出て、青いチェック・ボタンで整う*

![slint Todo](/images/gui-todo-slint.png)

最大の差は日本語表示でした。egui は既定フォントにCJKグリフを含まず、日本語が豆腐（□）になります。表示するには `FontDefinitions` でフォントを自前で登録する一手間が要ります。

```rust
// egui で日本語を出すための一手間
let mut fonts = egui::FontDefinitions::default();
let bytes = std::fs::read("/System/Library/Fonts/ヒラギノ角ゴシック W3.ttc")?;
fonts.font_data.insert("jp".into(), egui::FontData::from_owned(bytes).into());
fonts.families
    .entry(egui::FontFamily::Proportional)
    .or_default()
    .insert(0, "jp".into());
ctx.set_fonts(fonts);
```

これを入れれば egui でも日本語は出ます。

![egui Todo（フォント設定後）](/images/gui-todo-egui-jp.png)
*egui: フォントを埋め込めば日本語表示（完了項目は打ち消し線）*

一方 iced / slint は既定でシステムの日本語フォントを使い、整ったテーマも標準です。「Todoを作って」と頼んでそのまま日本語で見られる状態になるのは iced / slint です。tauri は HTML/CSS なので、整えやすさは普段のWeb制作の資産がそのまま使えます。

# ビルドが通った後の検証ループを誰が閉じるか

ここまでは人間が画面を見て整っているかを確かめてきました。しかしAIに書かせて回す前提では、もう一段の問いが要ります。「追加」を押したら本当に増えるのか、チェックで残数が減るのか、ドラッグで並べ替わるのか。それを目視に頼らず、自動テストやAI自身が繰り返し確かめられるかです。

Webバックエンドなら、AIは「サーバを起動 → `curl` で叩く → 返ってきたJSONを読む → 直す」で修正ループを自分で閉じられます。入力と出力がともにテキストなので、AIが完全にループの中に居られます。ところがGUIは、コンパイルが通っても「ボタンが効くか、レイアウトが崩れていないか、日本語が豆腐になっていないか」は何も保証されません。第3回で見た silent failure（型では気づけずテストが要る）の、GUI版です。ビルドの緑は必要条件にすぎず、残りのループを閉じる手段がcrateごとにどれだけ用意されているかが、AIに任せたときの開発しやすさを分けます。

そこで4つのcrateそれぞれで同じTodo（入力、追加、完了チェック、削除、残数、そしてドラッグでの並べ替え）を実装しました。各crate公式のテストハーネスで「ビルド後の検証ループが閉じるか」を実測しました（バージョンは本文記載と同一、macOS arm64）。

| | egui | iced | slint | tauri |
|---|---|---|---|---|
| 公式ハーネス | egui_kittest | iced_test | i-slint-backend-testing | tauri-driver |
| 要素の特定 | AccessKitツリー（ラベル/ロール） | 表示テキスト | accessible-label | DOM（role/text） |
| クリック・入力 | できる | できる＋`update`純粋関数 | できる（同期API） | できる |
| ドラッグ | 低レベルイベント注入で書ける | 生イベント＋工夫が要る | 専用`mock_drag`で一発 | WebDriver Actionsで得意 |
| 見た目（スナップショット） | wgpuで可 | 可 | 標準テストAPIに無し | Web資産で可 |
| ヘッドレス（macOS） | 可 | 可 | 可 | フロントのみ可／WebView越しは不可 |

## 基本操作とドラッグで分かれる

基本操作（クリック、入力、チェック、削除）は4つとも要素を特定して駆動でき、ループは閉じました。要素の特定の仕方に各思想が出ます。egui と slint はアクセシビリティツリー（スクリーンリーダが読むのと同じ意味の木）を辿り、iced は表示テキストで、tauri は DOM を見ます。

意味の木は `get_by_label("追加")` のように人間にもAIにも読める形です。実際 egui では、既定フォントに日本語が無く画面は豆腐でも、`get_by_label("追加")` は成功しました。ピクセルは壊れていても意味の木は正しいわけです。だからこそ意味の木での検証（要素が在るか）とスナップショット（見た目が壊れていないか）は別物で、両方要ります。

差が出たのは、ユーザー操作として複雑なドラッグでした。ドラッグは「押す→（複数地点を）動かす→離す」という連続したポインタ列で、単発のクリックより一段難しい操作です。

- **slint** は専用APIがあります。`mock_drag(target, button)` の一行で、押下→補間しながら移動→解放までを同期で発火します（テストcrate自身がこのAPIで内部テストを書いています）。最も素直です
- **egui / iced** はドラッグ専用ヘルパーが無く、低レベルのポインタイベントを手で組み立てます

```rust
// slint: 専用APIで「押下→移動→解放」を一行
handle.mock_drag(target, PointerEventButton::Left);

// egui: 専用ヘルパーは無く、低レベルのポインタイベントを手で流す
harness.drag_at(start);  harness.run();   // 押下
harness.hover_at(mid);   harness.run();   // 移動（複数回）
harness.hover_at(drop);  harness.run();
harness.drop_at(drop);   harness.run();   // 解放
```

iced に至っては `simulator` にドラッグAPI自体が無く、生のマウスイベント列を `simulate()` に渡します。tauri は DOM なので、WebDriver の Actions API（`pointerDown→pointerMove→pointerUp`）や Playwright の `dragTo` がそのまま使え、ドラッグはむしろ得意分野です。

そしてドラッグは、AIが一発で書くと静かに外す典型でもありました。実測でこう転びました。

- egui で行全体をドラッグ元にしたら、中のチェックボックスと削除ボタンのクリックが効かなくなりました。ドラッグ領域が下のウィジェットの当たり判定を奪うためで、コンパイルは通り、テストで初めて露見します。直すには、つまみ（ハンドル）だけをドラッグ元にします
- iced の `simulate()` はバッチ内の全イベントを「最後に指定したカーソル位置」で当たり判定する仕様で、押下→移動→解放を素直に1回で流すとカーソルが始点に貼り付き、移動が検知されず並べ替えが無言で起きません（テストは「成功」して見えて何もしていない）。押下と解放でカーソル位置を入れてフェーズを分ける必要があります

どちらも「コンパイルは通る、テストは緑に見える、でも実は動いていない」で、GUI版の silent failure です。AIに「ドラッグのテストを書いて」と頼んで返ってきた緑を、そのまま信じてはいけない例です。

なお、いちばん安い検証は GUI を起動しないテストです。iced の Elm Architecture は `update` が純粋関数なので、画面なしで状態遷移を直接確かめられます（実測では0.00秒）。

```rust
// iced: update は純粋関数。GUIを起動せず状態遷移をテストできる
let mut state = State::default();
update(&mut state, Message::InputChanged("牛乳を買う".into()));
update(&mut state, Message::AddTask);
assert_eq!(state.remaining(), 1);   // コンパイラとテストだけでループが閉じる
```

## 時間駆動の完了アニメをテストする

ドラッグの次は、時間で動くものです。各アプリに「タスク完了時に紙吹雪が舞うお祝いアニメ」（約800msのバースト）を足し、実装とテストにどれだけコードがかかるかを実測しました。まずは、できあがったお祝いアニメの途中フレームです。スクショの撮り方が crate ごとに違う点にも注目してください。

![egui: 完了で紙吹雪が舞う（ダークテーマ）](/images/gui-confetti-egui.png)
*egui:「記事を書く」を完了にした瞬間。egui_kittest の wgpu 描画で途中フレームを撮影（日本語はフォント登録済み）*

![slint: 完了で紙吹雪が舞う（ライトテーマ）](/images/gui-confetti-slint.png)
*slint:「ケーキを焼く」を完了にした瞬間。標準テストAPIにスクショが無いので、ソフトウェアレンダラ＋モック時間で撮影*

![tauri: 完了で紙吹雪が舞う（DOM/canvas）](/images/gui-confetti-tauri.png)
*tauri: フロント（DOM）を Playwright の `page.clock` で t≈250ms に固定して撮影*

| | 実装 | テスト | 仮想時間を進められるか |
|---|---|---|---|
| slint | **約67行**（最小） | 約124行 | `mock_elapsed_time()` で可 |
| egui | 約126行 | 約184行 | `with_step_dt`＋`run_steps` で可 |
| iced | 約124行 | 約169行 | ハーネス不可 → 純粋関数化で対応 |
| tauri（フロント） | 約141行 | 約85行 | Playwright `clock.runFor` で可 |

まず、アニメはテストするコード量が実装と同じくらい要ります（実装67〜141行に対しテスト85〜184行）。宣言的に `animate` を書ける slint が実装最小です。

時間駆動アニメをテストできるかは、ハーネスが仮想時間を決定論的に進められるかで決まります。

- **slint** はテストバックエンドに `mock_elapsed_time(Duration)` があり、これで宣言的アニメの補間値そのものが進みます。仮想時間を進めて途中値→終了値を読むだけです
- **egui** は `with_step_dt` で1フレームの経過時間を固定し、`run_steps(n)` で n フレーム進めます
- **tauri** は Playwright の `page.clock` で時間を握れます
- **iced だけ** simulator に時計が無く、`window::frames()` サブスクリプションも回りません（生のリドロー要求を流してもメッセージは0でした）。アニメ進行を「経過時間を引数に取る純粋関数」に切り出して初めてテストできます。さきほどの「純粋関数ならGUIなしでテストできる」が、時間軸でもそのまま使えるわけです

```rust
// slint: 仮想時間を決定論的に進めて、宣言的アニメの途中値を読む
toggle_done(&app, 0);                            // 完了 → 紙吹雪発火
mock_elapsed_time(Duration::from_millis(400));   // 仮想時間を 400ms 進める
let p = app.get_burst_progress();
assert!(p > 0.0 && p < 1.0);                      // アニメ途中
mock_elapsed_time(Duration::from_millis(600));   // さらに 600ms（計 1000ms）
assert_eq!(app.get_burst_progress(), 1.0);       // 終了
```

そして4つとも、「コンパイルは通る、一見動く、でも実は検証できていない」というアニメ固有の落とし穴を踏みました。

- **egui**: `run()` は再描画要求が続く限り回るので、アニメ中はフレーム上限超過でパニックします。`run_steps` で明示的にフレームを刻みます
- **iced**: simulator がフレームを進めないので、アニメのテストは純粋関数に切り出さないと一行も書けません
- **slint**: 1つのハンドラで進行値を `0 → 1` と続けて代入すると、中間値0が観測されず補間されずに即1.0になります。ゲート用プロパティを挟んで「0」を確定させる一手間が要ります
- **tauri**: `clock.fastForward(800)` は requestAnimationFrame を1回しか撃ちません（終了状態へ一足飛び）。途中フレームを見るには `clock.runFor(800)`（60fps相当でフレーム送り）が要ります

時間という軸が入ると、「緑なのに動いていない」が一段巧妙になります。時間駆動アニメは silent failure のなかでも捕まえにくいです。閉じるにはハーネスの時間制御（slint/egui/tauri）か、時間依存を純粋関数へ追い出す設計（iced）の、どちらかが要ります。

## tauri の検証ループは macOS で閉じない

tauri は「DOMだからWeb自動化の全資産が使える」という理屈が正しく、実際フロントエンドのドラッグは Playwright でテストできました。ところが、アプリ（WebView）越しにE2Eする公式の `tauri-driver` は Windows と Linux にしか対応していません。macOS には WKWebView を駆動する WebDriver が無いためです。実際に macOS で `cargo install tauri-driver` を実行すると、ビルドとインストールはどちらも問題なく成功します。しかし実行すると、即座に「このプラットフォームには対応していません」という1行を出して終了します。CLI/server/webdriver の各モジュールが linux/windows 限定の `cfg` で丸ごと除外され、macOS向けにはこの1行を返すだけのフォールバックしか残らないためです。

```
$ cargo install tauri-driver --locked
   Installed package `tauri-driver v2.0.6` (executable `tauri-driver`)
$ tauri-driver --help
tauri-driver is not supported on this platform
```

これは第2回の `libpq`、本記事の gtk4 とは少し違う止まり方です。ビルドは通るのに、実行すると何もしないまま終わるという、ここでも silent failure に近い構図です。AIが「tauri-driver なら macOS でもE2Eできる」と記憶で書けば、インストールまでは進んでも、テストは何ひとつ実行されないまま終わります。回避するには、アプリ越しのE2Eを諦めてフロントエンド（DOM）だけをブラウザでテストするか、コミュニティの WKWebView ドライバ（tauri-plugin-webdriver 等）を試すことになります。

検証ループの実測結果をまとめておきます（各crate公式ハーネスで同一仕様のTodo＋ドラッグ並べ替えをテスト / macOS arm64）。

- **egui**（egui_kittest 0.34）: AccessKitツリーで要素特定。基本操作・ドラッグ・スナップショットすべて green（計7テスト）。基本操作はGPU不要、スナップショットのみ wgpu(Metal)
- **iced**（iced_test 0.14）: `update()` の純粋単体テスト（GUI不要）10件＋`simulator` のUI駆動7件 green。ドラッグは生イベント列＋フェーズ分割が必須
- **slint**（i-slint-backend-testing 1.16）: 完全ヘッドレス・同期API・並列実行可で6件 green。ドラッグは専用 `mock_drag`。ただし `SLINT_EMIT_DEBUG_INFO=1` が無いと要素が見つからない点に注意。標準テストAPIにスクショは無い
- **tauri**（2.11）: フロントエンドは Playwright でドラッグ含め5件 green。だが公式 `tauri-driver` は macOS非対応（ビルドは通るが実行時に終了）で WebView越しE2E は閉じない
- **gtk4**: 第2回・本記事のとおりそもそもビルドできず、検証以前

## 検証ループを閉じる5つの定石

GUIの検証ループを閉じるための、crate横断の定石を5つ挙げます。

1. **ロジックを純粋関数に寄せる。** iced の `update()` のように、状態遷移を `(状態, メッセージ) → 状態` の純粋関数にすれば、GUIを一切起動せず `#[test]` で検証できます。最も安く確実なループです。第2回からの「adapterで隔離（撤退可能性）」は、そのままテスト容易性になります
2. **アクセシビリティツリーをAIが読む面にする。** egui/slint のテストは、スクリーンリーダ向けの意味の木（ラベル、ロール、値）を辿ります。アクセシブルに作ることが、そのままテスト可能でAI操作可能にすることです。Pure Rust の横断基盤 [AccessKit](https://github.com/AccessKit/accesskit)（egui の kittest もこれ）がその共通言語になりつつあります
3. **スナップショット（ゴールデン画像）で見た目を守る。** 意味の木では「日本語が豆腐」を捕まえられません。ヘッドレスで描画してPNGを基準画像と比較すれば、レイアウト崩れやフォント欠けを検出できます（egui_kittest/iced_test は標準装備、slint は標準テストAPIには無い）。画像比較は壊れやすいので、公式も「可能なら通常のテストを優先」と注意しています。スクリーンショットを画像入力できるLLMに渡して「日本語が出ているか」を判定させる手も現実的になってきました
4. **WebViewはWebの検証資産を丸ごと継承する。** tauri（や dioxus-desktop）は DOM なので、Playwright/WebDriver がそのまま使え、ドラッグも Actions API で書けます。代償は、ネイティブ用ドライバのプラットフォーム差（macOSの穴）と実行環境の重さです
5. **最後の手段は computer-use（画面＋視覚）。** 画面を見てマウスを動かせるAIなら、gtk4 だろうと任意のGUIを人間同様に操作できます。どんなcrateでも使える反面、遅く、壊れやすく、ディスプレイが要ります。他の手が無いときの最終手段です

# 用途で割り切る推奨

GUIは用途で正解が変わります。AIに書かせる前提では、次の整理が実務的です。

- ツール、可視化、社内用の素早いGUI → **egui**（即時モードで単純、Pure Rustで環境を選ばない。ただし日本語表示にはフォント設定が要る）
- きちんと状態管理したいデスクトップアプリ、整った日本語UIを最小手数で → **iced**（Elmアーキテクチャで規律が出て、既定で整う）
- 組込み、独自デザインシステム、既定の見た目を重視 → **slint**（宣言的DSL。既定テーマが最もモダン。ただしライセンス3択に注意）
- Webフロント資産がある、配布を重視 → **tauri**（Web技術＋小さいバイナリ。段取りは多い）
- 既存のGTK資産がある時だけ → **gtk4**（C依存の環境構築コストを許容できる場合）

共通して、どれも一発書きは外す前提で、コンパイラのエラーを返して修正させる運用にすることです。最新版を使うなら、AIに「このバージョンで」と明示し、エラーが出たら臆せず投げ返すのが速いというのが実測からの教訓です。

# 連載総括

全4回を振り返ると、crate選定とは、AIが「書く→ビルド→修正」を回すとき、どこで止まり、どこで守られるかを選ぶことでした。

| 回 | テーマ | AIが「止まる / 守られる」ポイント |
|---|---|---|
| 1 | デファクト群 | 学習データが厚く、そもそも止まりにくい。デファクトを外す時だけ理由を問う |
| 2 | Web / DB | コンパイル時検証（sqlx/diesel）がスキーマ不整合を弾く＝レビューを代わってくれる |
| 3 | 時刻 | TZ/DSTの意味論は型で守れるが、`since`の向きのような取り違えはテストが要る |
| 4 | GUI | ビルドが通る環境を作れるか。新APIでも復旧ループが回るか |

連載全体の結論は3つです。

1. **「一発書きの精度」だけで選ばない。** 一発目を外しても、コンパイラやビルドが具体的なエラーを返して修正ループが1〜2回で収束するcrateが、実務では強い。第2回のDB、第4回のGUIで繰り返し確認できました
2. **C依存や外部依存はAIエージェント最大の足かせ。** 第2回の libpq、第4回の GTK は同じ構図です。Pure Rust なら環境構築ごとAIに任せられます
3. **人間レビュアーの役割は「観点で問う」ことに移った。** コードの正しさはコンパイラと修正ループに委ね、人間は4観点（とりわけ立ち上げと反復速度、エコシステム適合、撤退可能性）で「なぜそれを選んだのか」を問います。Rustを普段書かない人がAIのライブラリ選定をチェックする、いちばん費用対効果の高い関わり方だと考えています

第1回の冒頭で、「AIで自分用のツールを作れるようになったが、ライブラリ選定の良し悪しが分からない」という不安から始めました。全部を理解できなくても、AIの修正ループがどこで止まるかを見抜く観点を持ち、要所で「なぜ」を問えば、選定はチェックできます。

以上、全4回でAIが行ったcrate選定をチェックする観点を紹介しました。AIと一緒にRustを書く皆様の助けになれば幸いです。
