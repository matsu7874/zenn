連載最終回は **GUI crate** です。これまで（[第1回](#)デファクトcrate群、[第2回](#)Web/DB、[第3回](#)時刻）と違い、GUIは正解の選択肢が用途で大きく変わる、もっとも割れる領域です。

それでも、AIに書かせる前提で選定基準を立てると、見えてくる景色は単純です。GUIのcrate選定は次の2点でほぼ決まります。

1. ビルドが通る環境を作れるか（Pure Rust か、外部のCライブラリや WebView が要るか）
2. AIの修正ループが回るか（GUIは進化が速く一発では当たらない。外したときに修正できるか）

実際に最新版へ書かせて検証すると、即時モードで「AIと相性が良い」とされる egui ですら、一発書きは 0/5 で全滅でした。理由は後述しますが、これはGUIを一発書きの精度で選んではいけないことを示しています。

# 4つの観点（GUI版）

第2回から使っている4観点を再掲します。見るのは、AIに書かせたときに回りやすいかです。

| 観点 | チェック内容 | NG例 |
|---|---|---|
| AI親和性 | AIが現行APIでコードを書け、エラーを渡せば修正できる | 旧APIを出し、しかも原因が環境で修正できない |
| 立ち上げと反復速度 | READMEからシンプルに動き、build と `cargo check` の待ちが短い | C依存・WebView前提で詰まる／依存が重くビルドが長い |
| エコシステム適合 | tokio/serde等と自然に繋がる | 独自世界観が強すぎる |
| 撤退可能性 | adapterで隔離でき、メンテが止まっても捨てられる | UI構造がアプリ全体に染み込む |

GUIで特に効くのは、立ち上げと反復速度（C依存や外部依存でビルドが通るか）と AI親和性（学習データがそのバージョンに追いついているか、外したときコンパイラが導いてくれるか）の2つです。GUIならではの観点として、ビルドが通った後にAIや自動テストで動作の検証ループを閉じられるかも効きます（後述）。

# GUIは設計思想で4つに分ける

GUIライブラリは数が多いので、設計思想の代表でまず4つに整理します。これにC依存の教材として gtk4 を加えます。

| パラダイム | crate | 描き方 |
|---|---|---|
| 即時モード | [egui](https://crates.io/crates/egui)（[eframe](https://crates.io/crates/eframe)） | 毎フレームUIを記述。状態管理が単純 |
| Elm Architecture | [iced](https://crates.io/crates/iced) | state / Message / update / view に分ける |
| 宣言的DSL | [slint](https://crates.io/crates/slint) | 独自の `.slint` 言語でUIを書きコンパイル |
| Web技術 | [tauri](https://crates.io/crates/tauri) | HTML/CSS/JS + Rust。OSのWebViewを使う |
| ネイティブ(C) | [gtk4](https://crates.io/crates/gtk4) | GTK4へのバインディング |

※バージョンは eframe/egui 0.34 / iced 0.14 / slint 1.16 / tauri 2.11 / gtk4 0.11（2026年6月時点）

ここでも見るのは「描画の美しさ」ではなく「AIに書かせる前提の回りやすさ」です。同じカウンタ（+1 / -1 ボタンと現在値）を各crateで一発書きさせ、`cargo build` の通過で評価しました。

## egui（eframe）の即時モード

egui は「毎フレーム、いまのUIを全部記述する」即時モードです。状態のツリーを管理しなくてよく、AIとの相性が良いと一般に言われます。

```rust
// eframe 0.34（2026年時点）の正しい書き方
use eframe::egui;

struct CounterApp { count: i64 }

impl eframe::App for CounterApp {
    fn ui(&mut self, ui: &mut egui::Ui, _frame: &mut eframe::Frame) {
        ui.label(format!("Count: {}", self.count));
        if ui.button("+1").clicked() { self.count += 1; }
        if ui.button("-1").clicked() { self.count -= 1; }
    }
}

fn main() -> eframe::Result<()> {
    eframe::run_native("Counter", eframe::NativeOptions::default(),
        Box::new(|_cc| Ok(Box::new(CounterApp { count: 0 }))))
}
```

ところが、このコードには罠があります。多くの学習データにある `fn update(&mut self, ctx, frame)` 形式は、eframe 0.34 ではもうコンパイルが通りません（必須メソッドが `ui` に刷新されました）。これが後述の「0/5」の原因です。

## iced の Elm Architecture

iced は state / Message / update / view に分ける、Elm由来のアーキテクチャです。状態遷移が型（`Message` enum）で表現され、規律ある設計ができます。

```rust
// iced 0.14 の正しい書き方（application の第1引数が状態初期化関数）
fn main() -> iced::Result {
    iced::application(Counter::default, Counter::update, Counter::view)
        .title("Counter")
        .run()
}
```

iced も罠があります。0.14 で `application` の第1引数が「タイトル文字列」から「状態初期化関数」に変わりました。`iced::application("Counter", ...)` という旧来の形は通りません。

## slint の宣言的DSL

slint は独自の `.slint` 言語でUIを宣言的に書き、ビルド時にネイティブコードへコンパイルします。`slint!` マクロを使えば Rust ファイル内に直接書けます。

```rust
slint::slint! {
    import { Button, VerticalBox } from "std-widgets.slint";
    export component MainWindow inherits Window {
        // ... プロパティとコールバック
    }
}
```

独自DSLなので学習データは相対的に薄いはずですが、後述のとおり一発書きは健闘しました。なお slint はライセンスが3択（Royalty-Free / GPLv3 / 商用）で、配布形態によっては自分のコードのライセンスに影響する点は選定上の注意です。

## tauri は Web技術と OS WebView を組み合わせる

tauri はフロントエンドをHTML/CSS/JSで書き、RustバックエンドとOSのWebView（macOSなら WKWebView）を組み合わせます。フロントの学習データは膨大で、バイナリにブラウザエンジンを同梱しないため小さくなります（Electronとの大きな違い）。

```rust
#[tauri::command]
fn increment(value: i64) -> i64 { value + 1 }

fn main() {
    tauri::Builder::default()
        .invoke_handler(tauri::generate_handler![increment])
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
```

ただし tauri は最小構成でも段取りが多い（後述）。

# 4つの観点で評価する

最新版で実際に書かせて採点した結果（下の実測メモ）を反映した評価です。

| 観点 | egui | iced | slint | tauri | （gtk4） |
|---|---|---|---|---|---|
| AI親和性 | △ 一発0/5（新API未追従）。だがヒントで1ラウンド復旧 | △ 一発0/5（新API未追従）。ヒントで1ラウンド復旧 | ○ 4/5。DSLエラーも1ラウンドで復旧 | ○ Web技術は学習厚く、Rust側は修正できる | △ コードは機械的だが、原因がC依存で環境ループ |
| 立ち上げと反復速度 | ◎ Pure Rustで素で通る・eframe雛形で即（208依存） | ◎ Pure Rustで素で通る・雛形あり（191依存） | ○ Pure Rustで通る・`slint!`で完結（335依存でやや重い） | △ CLI/設定/アイコン/HTML/WebView(Xcode)一式が要る（272依存） | ✗ pkg-config+GTK必須・素では一歩も進めない |
| エコシステム適合 | ○ | ○ tokio等 | △ DSL世界観 | ○ Web資産/tokio | △ GTK世界観 |
| 撤退可能性 | ○ 隔離しやすく開発も活発 | △ MVU構造が染みる（開発は活発） | △ DSL層が挟まる（開発は活発） | ○ フロント/バック分離で剥がしやすい | △（開発は活発） |
| 検証ループの閉じやすさ | ○ a11yツリーで駆動・スナップショット可。ドラッグは低レベル注入 | ○ `update`純粋関数が強い＋`simulator`。ドラッグは生イベント＋要工夫 | ◎ 専用ハーネスでドラッグも一発・完全headless | ○ DOMでWeb資産。ただしWebView越しE2EはmacOS非対応 | ✗ ビルド不可で検証以前 |

※ 一発書きの精度を実測したのは、単一バイナリで完結する egui / iced / slint です。tauri（Rust＋Web＋設定の複合）と gtk4（環境構築で止まる）は一発書きの粒度に合わないため、ビルド、依存、環境の観点で評価しています。

※ 最終行「検証ループの閉じやすさ」は、4観点に加えたGUI固有の軸です（ビルドが通った後、自動テストやAIで動作を確かめられるか）。各crateで実際にテストハーネスを動かした実測は、後述の「ビルドが通っても終わらない」節で扱います。

## AI親和性の実測

学習データの所在から、当初は「egui（即時モードで単純）と tauri（Web技術で学習データ豊富）が一発書きに強い」と予想していました。実測は予想を裏切りました。

- **egui: 0/5**。5試行すべてが、eframe 0.34 で廃された旧 `fn update` 形式で書き、`E0046 missing ui` でコンパイルエラー
- **iced: 0/5**。5試行すべてが、0.14 で変わった `application` の第1引数を旧来のタイトル文字列で渡し、`E0277 &str: BootFn` でコンパイルエラー
- **slint: 4/5**。独自DSLながら健闘。失敗1件も `std-widgets` の import 漏れ

GUIは進化が速く、最新版でAPIが変わると、学習データが追いつかず一発では当たりません。「即時モードだからAIと相性が良い」という一般論は、バージョンが動くと崩れます。

ただし、失敗は全てコンパイルエラーでした（第3回の jiff のような silent failure はゼロ）。コンパイラのエラー出力だけを返して再修正させると、次のようになります。

:::message
**実測メモ**（Claude Opus 4.8 / egui・iced・slint 各5試行 / ドキュメント参照を禁じた「一発書き」/ `cargo build` の通過で判定）

- 一発書き合格: **egui 0/5・iced 0/5・slint 4/5**
- 失敗の主因: egui = eframe 0.34 で `App` の必須メソッドが `update` → `ui` に刷新／iced = 0.14 で `application` の第1引数がタイトル文字列 → 状態初期化関数に破壊的変更／slint = `std-widgets` の import 漏れ1件。**全てコンパイルエラー（silent failure はゼロ）**
- 修正ループ: 失敗11件すべて、**コンパイラの出力（`help:` 行を含む）だけを返したら1ラウンドで全て build 成功**。egui の `help: implement the missing item: fn ui(...)` が決定打になった
:::

一発書きの精度だけ見れば egui/iced は最低ですが、修正ループは1ラウンドで全て収束しました。第2回や第3回と同じく、一発書きの精度ではなく**復旧ループの回りやすさ**で評価すべきだということが、GUIでも再現したわけです。

# ビルドと配布がGUI選定を支配する

GUI選定で最も効くのがこの観点です。実測した依存数、クリーンビルド時間、バイナリサイズを示します。

| crate | ユニーク依存数 | release クリーンビルド | release バイナリ | 外部依存 |
|---|---|---|---|---|
| iced | 191 | 20.4s | 10.2 MB | Pure Rust |
| egui (eframe) | 208 | 22.2s | 11.5 MB | Pure Rust |
| tauri | 272 | 28.5s | **8.1 MB** | OS WebView・要Xcode |
| slint | 335 | 32.7s | **8.4 MB** | Pure Rust |

時刻crate（第3回は依存2〜15）と比べ、GUIは 191〜335依存 と桁違いに重く、クリーンビルドは20〜33秒、バイナリは8〜11.5MBです。バイナリは tauri / slint が小さめです。tauri はOSのWebViewを使うためブラウザを同梱せず、Web技術系でも肥大しません。

## gtk4 のC依存の壁

第2回で diesel が `ld: library 'pq' not found` でリンク失敗したのと同じ構図が、GUIではより大規模に起きます。pkg-config も GTK4 も入っていない素のmacOSで gtk4 をビルドすると、次のようになります。

```
warning: glib-sys@0.22.6: Could not run `pkg-config --libs --cflags glib-2.0 ...`
  The pkg-config command could not be found.
error: failed to run custom build command for `glib-sys`
（gobject-sys / gio-sys / gdk-pixbuf-sys / pango-sys / cairo-sys-rs /
  graphene-sys / gdk4-sys / gsk4-sys / gtk4-sys も同様に失敗）
```

10個の `*-sys` crateが軒並み pkg-config と GTK 一式を要求して全滅します。AIエージェントにとって、これはコードの修正では解決せず、環境構築のループに迷い込むパターンです。一方、egui / iced / slint は Pure Rust なので、`cargo build` が通る環境さえあれば環境構築ごとAIに任せられます。

## tauri は設定とアセットの段取りで中間に位置する

tauri は依存こそ Pure Rust 寄りですが、最小のカウンタをビルドするだけでも、アイコン画像、`tauri.conf.json`、`build.rs`、フロントエンドのHTML、OS WebView(Xcode) が必要です。実際、本検証でも「`dist` が無い」「`icon.png` が無い」で2回足踏みしてからビルドが通りました。`cargo tauri` のような専用CLIで雛形を生成するのが前提になります。初速の観点では、Pure Rust の egui/iced/slint に一歩譲ります。

# ちょっとしたTodoで見る「整いやすさ」

カウンタのような最小例では差が出ませんが、実用アプリでは、整ったものがどれだけ手早く作れるかが効きます。同じ仕様のTodo（テキスト入力、追加、完了チェック、削除、残数表示）を egui / iced / slint で書き、デフォルトの見た目を比べました。

| | egui | iced | slint |
|---|---|---|---|
| デフォルトテーマ | 素朴（ダーク） | 整ったライト | モダン（ネイティブ風） |
| 日本語表示 | **豆腐（□）になる** | デフォルトでOK | デフォルトでOK |
| 標準ウィジェット | 入力/チェック/ボタン一通り | 一通り＋装飾済み | 一通り＋装飾済み |

![egui Todo（日本語が豆腐）](/images/gui-todo-egui.png)
*egui: 既定フォントにCJKグリフが無く、日本語が □ になる*

![iced Todo](/images/gui-todo-iced.png)
*iced: 既定で日本語が出て、青いチェック・ボタンで整う*

![slint Todo](/images/gui-todo-slint.png)
*slint: モダンな既定テーマ。日本語もそのまま*

最大の差は日本語表示でした。**egui は既定フォントにCJKグリフを含まず、日本語が豆腐（□）になります**。表示するには `FontDefinitions` でフォントを自前で登録する一手間が要ります。

```rust
// egui で日本語を出すための一手間
let mut fonts = egui::FontDefinitions::default();
let bytes = std::fs::read("/System/Library/Fonts/ヒラギノ角ゴシック W3.ttc")?;
fonts.font_data.insert("jp".into(), egui::FontData::from_owned(bytes).into());
fonts.families
    .entry(egui::FontFamily::Proportional)
    .or_default()
    .insert(0, "jp".into());
ctx.set_fonts(fonts);
```

これを入れれば egui でも日本語は出ます。

![egui Todo（フォント設定後）](/images/gui-todo-egui-jp.png)
*egui: フォントを埋め込めば日本語表示（完了項目は打ち消し線）*

一方 iced / slint は既定でシステムの日本語フォントを使い、整ったテーマも標準です。「Todoを作って」と頼んでそのまま日本語で見られる状態になるのは iced / slint です。egui は即時モードで書きやすいものの、見た目を整えるにはフォントとテーマの設定が要ります。tauri は HTML/CSS なので、整えやすさは普段のWeb制作の資産がそのまま効きます。

整いやすさ（手早く、整った画面になるか）という観点では、**既定で整って日本語も出る iced / slint が有利**で、egui は一手間ぶん割り引くと評価できます。これも「AIに書かせて、どれだけ早くまともな画面になるか」という連載の視点の一部です。

# ビルドが通った後の検証ループを誰が閉じるか

ここまでは人間が画面を見て整っているかを確かめてきました。しかしAIに書かせて回す前提では、もう一段の問いが要ります。「追加」を押したら本当に増えるのか、チェックで残数が減るのか、ドラッグで並べ替わるのか。それを目視ではなく、自動テストやAI自身が繰り返し確かめられるかです。ここがGUIとWebバックエンドの決定的な違いです。

Webバックエンドなら、AIは「サーバを起動 → `curl` で叩く → 返ってきたJSONを読む → 直す」で修正ループを自分で閉じられます。入力も出力もテキストなので、AIが完全にループの中に居られます。ところがGUIは、コンパイルが通っても「ボタンが効くか、レイアウトが崩れていないか、日本語が豆腐になっていないか」は何も保証されません。これは第3回の **silent failure**（型では気づけずテストが要る）の、GUI版です。ビルドの緑は必要条件にすぎず、残りのループを閉じる手段がcrateごとにどれだけ用意されているかが、AIに任せたときの開発しやすさを分けます。

そこで4つのcrateそれぞれで同じTodo（入力、追加、完了チェック、削除、残数、そしてドラッグでの並べ替え）を実装し、各crate公式のテストハーネスで「ビルド後の検証ループが閉じるか」を実測しました（バージョンは本文記載と同一、macOS arm64）。

| | egui | iced | slint | tauri |
|---|---|---|---|---|
| 公式ハーネス | egui_kittest | iced_test | i-slint-backend-testing | tauri-driver |
| 要素の特定 | AccessKitツリー（ラベル/ロール） | 表示テキスト | accessible-label | DOM（role/text） |
| クリック・入力 | ◎ | ◎ ＋ `update`純粋関数 | ◎（同期API） | ◎ |
| **ドラッグ** | △ 低レベルイベント注入 | △ 生イベント＋要工夫 | ◎ 専用`mock_drag` | ◎ WebDriver Actions |
| 見た目（スナップショット） | ◎ wgpu | ◎ | ✗ 標準テストAPIに無し | ◎ Web資産 |
| ヘッドレス（macOS） | ◎ | ◎ | ◎ | フロントのみ◎／WebView越し✗ |

## 基本操作とドラッグで分かれる

基本操作（クリック、入力、チェック、削除）は4つとも要素を特定して駆動でき、ループは閉じました。要素の特定の仕方に各思想が出ます。egui と slint は**アクセシビリティツリー**（スクリーンリーダが読むのと同じ意味の木）を辿り、iced は表示テキストで、tauri は DOM を見ます。とりわけ意味の木は `get_by_label("追加")` のように人間にもAIにも読める形で、JSON APIレスポンスのGUI版にあたります。実際 egui では、既定フォントに日本語が無く画面は「□□」と豆腐でも、`get_by_label("追加")` は成功しました。ピクセルは壊れていても意味の木は正しいわけです。だからこそ意味の木での検証（要素が在るか）とスナップショット（見た目が壊れていないか）は別物で、両方要ります。

差が出たのは、ユーザー操作として複雑なドラッグでした。ドラッグは「押す→（複数地点を）動かす→離す」という連続したポインタ列で、単発のクリックより一段難しい操作です。ここでハーネスの地力が出ます。

- **slint** は専用APIがあります。`mock_drag(target, button)` の一行で、押下→補間しながら移動→解放までを同期で発火します（テストcrate自身がこのAPIで内部テストを書いています）。最も素直です。
- **egui / iced** はドラッグ専用ヘルパーが無く、低レベルのポインタイベントを手で組み立てます。

```rust
// slint: 専用APIで「押下→移動→解放」を一行
handle.mock_drag(target, PointerEventButton::Left);

// egui: 専用ヘルパーは無く、低レベルのポインタイベントを手で流す
harness.drag_at(start);  harness.run();   // 押下
harness.hover_at(mid);   harness.run();   // 移動（複数回）
harness.hover_at(drop);  harness.run();
harness.drop_at(drop);   harness.run();   // 解放
```

iced に至っては `simulator` にドラッグAPI自体が無く、生のマウスイベント列を `simulate()` に渡します。tauri は DOM なので、WebDriver の Actions API（`pointerDown→pointerMove→pointerUp`）や Playwright の `dragTo` がそのまま効き、ドラッグはむしろ得意分野です（実測でもフロントエンドのドラッグ並べ替えは通りました）。

そしてドラッグは、AIが一発で書くと静かに外す典型でもありました。実測でこう転びました。

- egui で行全体をドラッグ元にしたら、中のチェックボックスと削除ボタンのクリックが効かなくなりました。ドラッグ領域が下のウィジェットの当たり判定を奪うためで、コンパイルは通り、テストで初めて露見します。直すには、つまみ（ハンドル）だけをドラッグ元にします。
- iced の `simulate()` はバッチ内の全イベントを「最後に指定したカーソル位置」で当たり判定する仕様で、押下→移動→解放を素直に1回で流すとカーソルが始点に貼り付き、移動が検知されず並べ替えが無言で起きません（テストは「成功」して見えて何もしていない）。押下と解放でカーソル位置を入れてフェーズを分ける必要があります。

どちらも「コンパイルは通る、テストは緑に見える、でも実は動いていない」で、GUI版 silent failure です。AIに「ドラッグのテストを書いて」と頼んで返ってきた緑を、そのまま信じてはいけない例です。

なお、いちばん安い検証は GUI を起動しないテストです。iced の Elm Architecture は `update` が純粋関数なので、画面なしで状態遷移を直接確かめられます（実測では0.00秒）。

```rust
// iced: update は純粋関数。GUIを起動せず状態遷移をテストできる
let mut state = State::default();
update(&mut state, Message::InputChanged("牛乳を買う".into()));
update(&mut state, Message::AddTask);
assert_eq!(state.remaining(), 1);   // コンパイラとテストだけでループが閉じる
```

## 時間駆動の完了アニメをテストする

ドラッグの次は、時間で動くものです。各アプリに「タスク完了時に紙吹雪が舞うお祝いアニメ」（約800msのバースト）を足し、実装とテストにどれだけコードがかかるかを実測しました。まずは、できあがったお祝いアニメの途中フレームです。スクショの撮り方が crate ごとに違う点にも注目してください。

![egui: 完了で紙吹雪が舞う（ダークテーマ）](/images/gui-confetti-egui.png)
*egui:「記事を書く」を完了にした瞬間。egui_kittest の wgpu 描画で途中フレームを撮影（日本語はフォント登録済み）*

![slint: 完了で紙吹雪が舞う（ライトテーマ）](/images/gui-confetti-slint.png)
*slint:「ケーキを焼く」を完了にした瞬間。標準テストAPIにスクショが無いので、ソフトウェアレンダラ＋モック時間で撮影*

![tauri: 完了で紙吹雪が舞う（DOM/canvas）](/images/gui-confetti-tauri.png)
*tauri: フロント（DOM）を Playwright の `page.clock` で t≈250ms に固定して撮影*

| | 実装 | テスト | 仮想時間を進められるか |
|---|---|---|---|
| slint | **約67行**（最小） | 約124行 | ◎ `mock_elapsed_time()` |
| egui | 約126行 | 約184行 | ◎ `with_step_dt`＋`run_steps` |
| iced | 約124行 | 約169行 | △ ハーネス不可 → 純粋関数化 |
| tauri（フロント） | 約141行 | 約85行 | ◎ Playwright `clock.runFor` |

まず、アニメはテストするコード量が実装と同じくらい要ります（実装67〜141行に対しテスト85〜184行）。宣言的に `animate` を書ける slint が実装最小です。

時間駆動アニメをテストできるかは、ハーネスが仮想時間を決定論的に進められるかで決まります。

- **slint** はテストバックエンドに `mock_elapsed_time(Duration)` があり、これで宣言的アニメの補間値そのものが進みます。仮想時間を進めて途中値→終了値を読むだけです。
- **egui** は `with_step_dt` で1フレームの経過時間を固定し、`run_steps(n)` で n フレーム進めます。
- **tauri** は Playwright の `page.clock` で時間を握れます。
- **iced だけ** simulator に時計が無く、`window::frames()` サブスクリプションも回りません（生のリドロー要求を流してもメッセージは0でした）。アニメ進行を「経過時間を引数に取る純粋関数」に切り出して初めてテストできます。さきほど iced の `update` で見た「純粋関数ならGUIなしでテストできる」が、時間軸でもそのまま効くわけです。

```rust
// slint: 仮想時間を決定論的に進めて、宣言的アニメの途中値を読む
toggle_done(&app, 0);                            // 完了 → 紙吹雪発火
mock_elapsed_time(Duration::from_millis(400));   // 仮想時間を 400ms 進める
let p = app.get_burst_progress();
assert!(p > 0.0 && p < 1.0);                      // アニメ途中
mock_elapsed_time(Duration::from_millis(600));   // さらに 600ms（計 1000ms）
assert_eq!(app.get_burst_progress(), 1.0);       // 終了
```

そして4つとも、「コンパイルは通る、一見動く、でも実は検証できていない」というアニメ固有の罠を踏みました。

- **egui**: `run()` は再描画要求が続く限り回るので、アニメ中はフレーム上限超過でパニックします。`run_steps` で明示的にフレームを刻みます（または上限を上げます）。
- **iced**: simulator がフレームを進めないので、アニメのテストは純粋関数に切り出さないと一行も書けません。
- **slint**: 1つのハンドラで進行値を `0 → 1` と続けて代入すると、中間値0が観測されず補間されずに即1.0になります。ゲート用プロパティを挟んで「0」を確定させる一手間が要ります。
- **tauri**: `clock.fastForward(800)` は requestAnimationFrame を1回しか撃ちません（終了状態へ一足飛び）。途中フレームを見るには `clock.runFor(800)`（60fps相当でフレーム送り）が要ります。

時間という軸が入ると、「緑なのに動いていない」が一段巧妙になります。時間駆動アニメは silent failure のなかでも捕まえにくく、閉じるにはハーネスの時間制御（slint/egui/tauri）か、時間依存を純粋関数へ追い出す設計（iced）の、どちらかが要ります。

## tauri の検証ループは macOS で閉じない

tauri は「DOMだからWeb自動化の全資産が効く」という理屈が正しく、実際フロントエンドのドラッグは Playwright でテストできました。ところが、アプリ（WebView）越しにE2Eする公式の `tauri-driver` は Windows と Linux にしか対応していません。macOS には WKWebView を駆動する WebDriver が無いためです。実際に macOS で `cargo install tauri-driver` すると、実行時エラーですらなく、コンパイルエラーで落ちます（ドライバのバイナリ定義が linux/windows の `cfg` にしか無い）。

```
error[E0425]: cannot find value `DRIVER_BINARY` in this scope
error[E0599]: no method named `into_native_object` found ...
error: could not compile `tauri-driver`
```

これは第2回の `libpq`、本記事の gtk4 と同じく、環境がループを止める構図です。AIが「tauri-driver で macOS でもE2Eできる」と記憶で書けば、コンパイルすら通らず確実に外します。回避するにはアプリ越しのE2Eを諦めてフロントエンド（DOM）だけをブラウザでテストするか、コミュニティの WKWebView ドライバ（tauri-plugin-webdriver 等）に出る必要があります。

:::message
**検証ループの実測**（各crate公式ハーネスで同一仕様のTodo＋ドラッグ並べ替えをテスト / macOS arm64 / バージョンは本文記載と同一）

- **egui**（egui_kittest 0.34）: AccessKitツリーで要素特定。基本操作・ドラッグ・スナップショットすべて green（計7テスト）。ドラッグは専用ヘルパー無し→`drag_at`/`hover_at`/`drop_at` の低レベル注入。基本操作はGPU不要、スナップショットのみ wgpu(Metal)
- **iced**（iced_test 0.14）: `update()` の純粋単体テスト（GUI不要）10件＋`simulator` のUI駆動7件 green。ドラッグは生イベント列＋フェーズ分割が必須
- **slint**（i-slint-backend-testing 1.16）: 完全ヘッドレス・同期API・並列実行可で6件 green。ドラッグは専用 `mock_drag`。ただし `SLINT_EMIT_DEBUG_INFO=1` が無いと要素が見つからない罠。標準テストAPIにスクショは無い
- **tauri**（2.11）: フロントエンドは Playwright でドラッグ含め5件 green。だが公式 `tauri-driver` は **macOS非対応**（コンパイル不可）で WebView越しE2E は閉じない
- **gtk4**: 第2回・本記事のとおりそもそもビルドできず、検証以前
:::

## 一般的な解決策

GUIの検証ループを閉じるための、crate横断の定石を5つ挙げます。

1. **ロジックを純粋関数に寄せる。** iced の `update()` のように、状態遷移を `(状態, メッセージ) → 状態` の純粋関数にすれば、GUIを一切起動せず `#[test]` で検証できます。最も安く確実なループです。第2回からの「アダプタで隔離（撤退可能性）」は、そのままテスト容易性になります。薄いシェルの下に厚いテスト可能な核を置く形が効きます。
2. **アクセシビリティツリーをAIが読む面にする。** egui/slint のテストは、スクリーンリーダ向けの意味の木（ラベル、ロール、値）を辿ります。アクセシブルに作ることが、そのままテスト可能でAI操作可能にすることです。pure Rust の横断基盤 [AccessKit](https://github.com/AccessKit/accesskit)（egui の kittest もこれ）がその共通言語になりつつあります。連載を貫く「構造化されたものはAIに強い」が、GUIでも成り立ちます。
3. **スナップショット（ゴールデン画像）で見た目を守る。** 意味の木では「日本語が豆腐」を捕まえられません。ヘッドレスで描画してPNGを基準画像と比較すれば、レイアウト崩れやフォント欠けの視覚的な silent failure を検出できます（egui_kittest/iced_test は標準装備、slint は標準テストAPIには無い）。画像比較は壊れやすい（色の差で落ちる）ので、公式も「可能なら通常のテストを優先」と注意しています。さらに一歩進めて、スクリーンショットを画像入力できるLLMに渡して「日本語が出ているか/Todoらしく見えるか」を判定させる手も現実的になっています。
4. **WebViewはWebの検証資産を丸ごと継承する。** tauri（や dioxus-desktop）は DOM なので、Playwright/WebDriver/WebdriverIO がそのまま効き、ドラッグも Actions API で一級市民です。DOMはAIの学習量が最も多く、操作面として強いものです。代償は、ネイティブ用ドライバのプラットフォーム差（macOSの穴）と実行環境の重さです。
5. **最後の手段は computer-use（画面＋視覚）。** 画面を見てマウスを動かせるAIなら、gtk4 だろうと任意のGUIを人間同様に操作できます。どんなcrateでも効く反面、遅く、壊れやすく、ディスプレイが要り、意味的な手がかりが無いぶん不安定です。他の手が無いときの最終手段です。

# 用途で割り切る推奨

GUIは用途で正解が変わります。AIに書かせる前提では、次の整理が実務的です。

- ツール、可視化、社内用の素早いGUI → **egui**（即時モードで単純、Pure Rustで環境を選ばない。ただし日本語表示にはフォント設定が要る）
- きちんと状態管理したいデスクトップアプリ、整った日本語UIを最小手数で → **iced**（Elmアーキテクチャで規律が出て、既定で整う）
- 組込み、独自デザインシステム、既定の見た目を重視 → **slint**（宣言的DSL。既定テーマが最もモダン。ただしライセンス3択に注意）
- Webフロント資産がある、配布を重視 → **tauri**（Web技術＋小さいバイナリ。段取りは多い）
- 既存のGTK資産がある時だけ → **gtk4**（C依存の環境構築コストを許容できる場合）

そして共通して、どれも一発書きは外す前提で、コンパイラのエラーを返して修正させる運用にすることです。最新版を使うなら、AIに「このバージョンで」と明示し、エラーが出たら臆せず投げ返すのが速い、というのが実測からの教訓です。

# GUIは環境と修正ループで選ぶ

- GUIは進化が速く、最新版ではAIの一発書きが外れやすい（egui/iced とも 0/5）。「即時モードだから書きやすい」より「学習データがそのバージョンに追いついているか」が効く
- だが失敗は全てコンパイルエラーで、ヒントを返せば1ラウンドで復旧できた。一発書きの精度表だけで選ぶと判断を誤る
- 選定を支配するのはビルドが通る環境である。Pure Rust の egui/iced/slint は素で通り、gtk4 は GTK 一式が無いと一歩も進めない。tauri はその中間
- さらにGUIでは、ビルドが通った後の検証ループ（自動テストやAIで「実際に動くか」を確かめられるか）も効く。とくにドラッグのような複雑操作は、slint が専用ハーネスで一発、egui/iced は低レベル注入で書け、tauri は Web の検証資産を継承する（ただし WebView越しE2E は macOS非対応）

# 連載総括

全4回を振り返ると、一本の筋が通っていました。crate選定とは、AIが「書く→ビルド→修正」を回すとき、どこで止まり、どこで守られるかを選ぶことでした。

| 回 | テーマ | AIが「止まる / 守られる」ポイント |
|---|---|---|
| 1 | デファクト群 | 学習データが厚く、そもそも止まりにくい。デファクトを外す時だけ理由を問う |
| 2 | Web / DB | **コンパイル時検証**（sqlx/diesel）がスキーマ不整合を弾く＝レビューの肩代わり |
| 3 | 時刻 | TZ/DSTの意味論は型で守れるが、`since`の向きのような取り違えは**テスト**が要る |
| 4 | GUI | **ビルドが通る環境**を作れるか。新APIでも**復旧ループが回る**か |

ここから、連載全体を貫く3つの結論にたどり着きます。

1. **「一発書きの精度」だけで選ばない。** 一発目を外しても、コンパイラやビルドが具体的なエラーを返して修正ループが1〜2回で収束するcrateが、実務では強い。第2回のDB、第4回のGUIで繰り返し確認できました。
2. **C依存や外部依存はAIエージェント最大の足かせ。** 第2回の libpq、第4回の GTK は同じ構図です。Pure Rust なら環境構築ごとAIに任せられます。
3. **人間レビュアーの役割は「観点で問う」ことに移った。** コードの正しさはコンパイラと修正ループに委ね、人間は4観点（とりわけ立ち上げと反復速度、エコシステム適合、撤退可能性）で、AIの選定に「なぜそれを選んだのか」を問う。これが、Rustを普段書かない人がAIのライブラリ選定をチェックする、いちばん費用対効果の高い関わり方です。

第1回の冒頭で、「AIで自分用のツールを作れるようになったが、ライブラリ選定の良し悪しが分からない」という不安から始めました。その答えは、全部を理解することではありません。AIの修正ループがどこで止まるかを見抜く4つの観点を持ち、要所で「なぜ」を問うことです。それだけで、AIに任せたRustのコードは、ぐっと信頼できるものになります。
