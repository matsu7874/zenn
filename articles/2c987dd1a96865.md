前回（[AI時代のRust crate選定 (2) ３つのデータベース接続](https://zenn.dev/estie/articles/79aacb4bc57e69)）は、データベースアクセスを題材に「コンパイル時検証はレビューの代替になる」という観点を扱いました。AIが書いたSQLのスキーマ不整合を、コンパイラがレビュアーとして肩代わりしてくれる、という話です。

今回のテーマは時刻です。一見地味ですが、時刻処理は「それらしく書けているのに、実は間違っている」コードが生まれやすく、AIに書かせるうえで罠の多い領域です。そして選択肢は [chrono](https://crates.io/crates/chrono) / [time](https://crates.io/crates/time) / [jiff](https://crates.io/crates/jiff) の3つに割れています。

この記事の結論を先に言うと、時刻crateは次の2軸で見るべきだということです。

1. タイムゾーンやDST（夏時間）の意味論を、型やメソッドで表現できるか
2. AIが「いまのAPI」で書けるか（学習データの量と新しさ）

そして実際に3つのcrateへ書かせて検証すると、一発書きの精度はライブラリの普及度の順番（chrono > time > jiff）でしたが、それ以上に失敗の質が決定的に違うことが分かりました。先に要点だけ言うと、timeの失敗はコンパイルエラーで安全に止まり、jiffの失敗は**コンパイルを通り抜けて値だけ間違える**（silent failure）のです。

# 時刻処理でAIがやりがちな間違い

具体的なcrateの前に、時刻処理そのものの「踏みやすい罠」を押さえます。これは言語やcrateを問わず共通で、AIと人間のどちらも間違えるポイントです。

- **naive（タイムゾーンなし）と aware（タイムゾーンあり）の取り違え**: 「2026-06-15 12:00」という壁時計の文字列が、UTC基準とJST基準のどちらか分からないまま計算してしまう
- **「とりあえずローカル時刻」**: サーバーのタイムゾーン設定に依存し、本番とローカルで結果が変わる
- **DST（夏時間）を跨ぐ加減算**: 「1日後」と「24時間後」は異なる。夏時間の切り替え日に、壁時計上で「1日後」は物理的に23時間後や25時間後になる
- **月末の加算**: 「1月31日の1ヶ月後」は何日か。2月31日は存在しないので、2月28日（や29日）にクランプする必要がある

特に3つ目のDSTは直感に反します。例として、アメリカ東部（America/New_York）が夏時間に切り替わる2026年3月8日の午前1時30分を起点に考えます。

| 起点 | 演算 | 結果（壁時計） |
|---|---|---|
| 2026-03-08 01:30 (EST) | **暦の上で1日後**（カレンダー演算） | 2026-03-**09** 01:30 |
| 2026-03-08 01:30 (EST) | **物理的に24時間後**（絶対演算） | 2026-03-**09** **02:30** |

3月8日の午前2時に時計が1時間進むため、同じ「翌日」でも、壁時計を保つカレンダー演算では01:30、物理的な24時間後では02:30と、1時間ずれます。この区別をcrateがどう表現するかが、時刻crate選定の肝です。

# crateを評価する4つの観点

第2回と同じ4観点で評価します。「人間の書きやすさ」ではなく「AIに書かせると回りやすいか」を見る観点でした。

| 観点 | チェック内容 | NG例 |
|---|---|---|
| AI親和性 | AIが現行APIでコードを書け、エラーを渡せば修正できる | 古い/存在しないAPIを出し、しかも修正できない |
| 立ち上げと反復速度 | READMEからシンプルに動き、build と `cargo check` の待ちが短い | feature指定や別crate併用で詰まる／checkが遅い |
| エコシステム適合 | tokio/serde/[tracing](https://crates.io/crates/tracing)等と自然に繋がる | 独自世界観が強すぎる |
| 撤退可能性 | adapterで隔離でき、メンテが止まっても捨てられる | アプリ全体に染み込み、放置されても逃げられない |

時刻領域で特に効くのは次の3点です。

- **AI親和性**: 時刻処理は「APIの新しさ」がそのまま精度に出る。古くからあるcrateほどAIが書け、新しいcrateほど学習データが薄く外しやすい。しかも後述のとおり、時刻には「修正できるか」が型では効かない罠（silent failure）がある
- **撤退可能性**: この領域は「枯れている＝安全」が崩れた（後述。chronoは作者がメンテ縮小を表明）。ただしフリーライダーの立場では、隔離して捨てられるなら保守が細っても使える。「保守の弱さ」を「捨てやすさ」でカバーできるかが論点になる
- **立ち上げと反復速度**: IANAタイムゾーン（`Asia/Tokyo`等）をどう持つかで依存とビルド時間が大きく変わる

# 時刻crateの三本柱

| crate | スタイル | タイムゾーンの扱い | 一言で言うと |
|---|---|---|---|
| chrono | 老舗、固定オフセット中心 | `DateTime<Tz>` / `NaiveDateTime`。IANAは [chrono-tz](https://crates.io/crates/chrono-tz) 併用 | 学習データ最多。事実上の旧デファクト |
| time | `no_std` 対応、依存最小 | `OffsetDateTime`（固定オフセット）。IANAは [time-tz](https://crates.io/crates/time-tz) 併用 | 組込み向き。featureが細分 |
| jiff | [Temporal](https://tc39.es/proposal-temporal/) 準拠、タイムゾーンファースト | `Zoned` がIANA TZを内蔵 | [ripgrep](https://crates.io/crates/ripgrep)作者による新標準候補（1.0手前） |

※バージョンは chrono 0.4.45 / time 0.3.49 / jiff 0.2.28（2026年6月時点）

## chrono: 老舗、ただし「ソフトdeprecated」

chrono はもっとも広く使われてきたcrateです。`DateTime<Utc>` や `NaiveDateTime` といった型はRust界隈で広く知られ、AIの学習データも豊富です。

```rust
use chrono::{Utc, TimeZone, Datelike, Timelike};
use chrono_tz::Asia::Tokyo;

// UTC epoch秒 → JST の壁時計
let jst = Utc.timestamp_opt(epoch, 0).single().unwrap().with_timezone(&Tokyo);
println!("{}:{}", jst.hour(), jst.minute());
```

ただし注意点が1つあります。2026年初頭、chronoの保守者が [Issue #1768](https://github.com/chronotope/chrono/issues/1768) で、APIデザインの古さと改修コストを理由に今後メンテナンスを縮小し、後継として jiff を薦めると表明しました。即座に危険が生じるわけではありませんが、「枯れていて安心」という前提が揺らいでいる点は、保守の観点で割り引く必要があります。

## time: 依存最小、`no_std` 対応

time は標準ライブラリとの相互運用を重視し、`no_std` でも使える軽量なcrateです。組込みや依存を絞りたい場面で選ばれます。

```rust
use time::OffsetDateTime;
use time::format_description::well_known::Rfc3339;

let odt = OffsetDateTime::parse("2026-06-15T12:00:00Z", &Rfc3339).unwrap();
let epoch = odt.unix_timestamp();
```

注意点は2つあります。1つは featureフラグが細かく分かれていること（`formatting` / `parsing` / `macros` などが既定オフ）。もう1つは、**timeはIANAタイムゾーンを持たない**ことです。`Asia/Tokyo` のようなタイムゾーンを扱うには別crateの [time-tz](https://crates.io/crates/time-tz) を併用します。これが後述の通り、AIに書かせるうえでの躓きどころになります。

## jiff: Temporal準拠のタイムゾーンファースト

jiff は [BurntSushi](https://github.com/BurntSushi)（ripgrep / regex の作者）による新しいcrateで、JavaScriptの [Temporal](https://tc39.es/proposal-temporal/) 提案を下敷きにした設計です。最大の特徴は、タイムゾーン付きの `Zoned` 型が標準で IANA タイムゾーンを扱えること、そしてカレンダー演算と絶対演算をメソッドで明確に分けることです。

```rust
use jiff::{civil, ToSpan};

let z = civil::date(2026, 3, 8).at(1, 30, 0, 0).in_tz("America/New_York")?;
let cal = z.checked_add(1.day())?;    // 暦の1日後  => 2026-03-09 01:30
let abs = z.checked_add(24.hours())?; // 物理24時間後 => 2026-03-09 02:30
```

`1.day()`（カレンダー）と `24.hours()`（絶対）が別物として書き分けられ、DSTを跨ぐと結果が変わるという意味論がコードに表れます。本記事の実測でも、この区別は全試行が正しく扱えました。一方で、jiff はまだ 1.0 未到達（0.2系で破壊的変更の余地あり）で、学習データも薄く、後述の通り別の落とし穴があります。

# 4つの観点で三本柱を評価する

実際に Claude Opus 4.8 へ各crateで同じ8課題を書かせ、ビルドと値の検証で採点した結果（下の実測メモ）を反映した評価が次の表です。◎○△は「AIに書かせる前提での相対評価」です。

| 観点 | chrono | time | jiff |
|---|---|---|---|
| AI親和性 | ◎ 一発書き5/5、学習データ最多。そもそも落ちない | ○ 3/5。time-tzのトレイト作法で躓くが、コンパイラが具体的に弾いて修正できる | △ 2/5。新APIをsilentに誤り、**型では気づけずテストが要る** |
| 立ち上げと反復速度 | ○ ほぼPure Rust。`check` 2.97s | △ feature細分＋time-tzで依存15、`check` 4.63s | ◎ 依存2、Pure Rust、`check` 0.93sで最速 |
| エコシステム適合 | ◎ serde等に広く対応 | ○ serde対応、TZは別crate | ◎ serde/RFC9557、jiff-sqlx等 |
| 撤退可能性 | △ 隔離はしやすいが、作者がメンテ縮小とjiff推奨を表明 | ○ 隔離しやすく開発も活発 | ○ 隔離しやすい。活発だが **1.0未達** |

第2回ではDB3crateとも保守の心配はありませんでしたが、今回は chrono のメンテ縮小が撤退可能性に影を落とします。「枯れている＝安全」が成立しない領域だということです。とはいえ、後述のとおり隔離してあれば捨てて移ればよく、致命傷にはなりません。

以下、特に判断を左右する観点を順に見ていきます。

## AI親和性で分かれる失敗の質

学習データ量の仮説（chrono最多）どおり、一発書きの精度は chrono 5/5 > time 3/5 > jiff 2/5 でした。ただし選定で効くのは精度そのものより、失敗の質のほうです。

- **chrono**: `dt - dt`、`+ Duration`、`checked_add_months` など直感的なAPIで、5/5。失敗なし
- **time**: 失敗2件はどちらもコンパイルエラー。`Asia/Tokyo` への変換には time-tz の拡張トレイト（`use time_tz::OffsetDateTimeExt;`）が必要だが、import を落としていた。そのため `to_timezone` がメソッド解決できず弾かれた。間違った値は出ない（安全側）
- **jiff**: 失敗3件はコンパイルが通ったのに値を間違えた。「2つの日付の差」を求める課題で、新しめのAPI `Date::since` の差の向きを誤り、+28日であるべきところを -28日（符号が逆）で返した。3試行とも同じ間違いだった

第2回の「コンパイル時検証がレビューを肩代わりする」は、DST/タイムゾーンの意味論（カレンダー演算と絶対演算の区別）には効きます。jiffの `1.day()` / `24.hours()` のように、メソッドや型で意味が分かれていれば、取り違えはコンパイルや結果に表れるからです。しかし、`since` の向きのように「同じ型（`i64`）で符号だけ違う」取り違えは、**型では防げません**。テストを書いて値を検証しない限り、`cargo build` が通った時点では気づけないのです。

:::message
**実測メモ**（Claude Opus 4.8 / 各crate5試行 / ドキュメント参照を禁じた「一発書き」/ 8課題をビルドと値で判定）

- 一発書き合格: **chrono 5/5、time 3/5、jiff 2/5**
- 失敗の質: time（2件）は**コンパイルエラー**（time-tzのトレイト未import等）。jiff（3件）は**silent failure**（`Date::since` の差の符号を誤り -28 を返す。型は合うので通る）
- 修正ループ: 失敗5件すべて、エラー出力（time: コンパイルエラー / jiff: テストの値不一致）だけを返したら **1ラウンドで全て修正**できた。ただし jiff は**テストが無いと失敗に気づけない**点が time と異なる
:::

つまり jiff は「一発書きの精度が最も低く、しかも失敗がコンパイルをすり抜ける」ため、使うならテストとセットが前提になります。逆に言えば、テスト体制があれば、jiffの意味論の明快さ（DST安全な演算）と軽さは大きな武器です。

## 「枯れている＝安全」が崩れた領域での撤退可能性

通常、ライブラリ選定では「枯れていて広く使われている」ことが安心材料です。chronoはまさにそれでした。しかし2026年、chronoの保守者自身が [#1768](https://github.com/chronotope/chrono/issues/1768) でメンテ縮小と jiff 推奨を表明したことで、この前提が崩れました。

とはいえ、chronoが即座に動かなくなるわけではありません。膨大な既存資産があり、AIの学習データも最多です。一方の後継候補 jiff は、まだ 1.0 に達しておらず、破壊的変更の余地があります。つまり「老舗ながら先細り（chrono）」対「将来有望だが未成熟（jiff）」という、どちらも単純に手放しで薦めにくい構図です。

ここで効くのが撤退可能性の発想です。時刻処理は幸い、`OffsetDateTime` や `Zoned` といった型がアプリの各所に直接出てこないよう、変換と整形をadapterに寄せれば隔離できます。私たちはエコシステムのフリーライダー（crateを育てる側ではなく使う側）なので、「どのcrateが長期で生き残るか」を当てにいく必要はありません。隔離してあれば、chronoのメンテが細った時点で jiff に差し替えればよいわけです。「保守が完璧か」を悩むより「いざという時に捨てられる形で使う」ほうが、この揺れている領域では現実的です。だからこそ、AIがどのcrateを選んでも、隔離されているかをまず人間の手で確認する価値があります。

## 立ち上げと反復速度を左右する依存数とfeature

依存とビルド時間の実測値です（クリーンビルド、同一環境）。

| crate | 依存数（ユニーク） | C依存 | release ビルド | `cargo check` |
|---|---|---|---|---|
| jiff | **2** | なし（Pure Rust） | **1.89s** | **0.93s** |
| chrono | 10 | core-foundation-sys（OS標準。追加インストール不要） | 3.35s | 2.97s |
| time（単体） | 7 | なし | — | — |
| time + time-tz | **15** | libc | 5.98s | 4.63s |

jiffは依存2で最速、IANAタイムゾーン込みでこれです。chronoは中庸。timeは単体なら7と最小ですが、IANAタイムゾーンのために time-tz を足すと15へ倍増します。ビルドも最長になり、`cargo check` は4.63秒かかります（修正ループ1回あたりの単価として効いてきます）。

なお、いずれも第2回のdieselの `libpq` のような外部システムライブラリのインストールは不要です。この領域に「C依存でビルドが詰まる」壁はありません。

time はfeatureが細分されており、`OffsetDateTime::parse` には `parsing`、整形には `formatting` を、それぞれ明示的に有効化する必要があります。`format_description!` マクロを使うには `macros` も有効化が必要です。最小featureから始めると、AIが整形やパースを書いた瞬間にコンパイルエラーで足踏みします。

# 同じ「DST安全な加算」を3 crateで書く

最後に、本記事の山場である「カレンダー演算と絶対演算の区別」を、3つのcrateでどう書くか並べます。

```rust
// jiff: メソッドで意味論が分かれる（最も明快）
let z = jiff::civil::date(2026, 3, 8).at(1, 30, 0, 0).in_tz("America/New_York")?;
z.checked_add(1.day())?;    // 暦の1日後   => 03-09 01:30（壁時計を保つ）
z.checked_add(24.hours())?; // 物理24時間後 => 03-09 02:30（DSTで1時間ずれる）
```

```rust
// chrono: Days（カレンダー）と Duration（絶対）で区別。chrono-tz が必要
use chrono::{Days, Duration, TimeZone};
use chrono_tz::America::New_York;
let dt = New_York.from_local_datetime(&naive).single().unwrap();
dt + Days::new(1);        // カレンダー => 03-09 01:30
dt + Duration::hours(24); // 絶対     => 03-09 02:30
```

```rust
// time: IANA TZ を持たないので time-tz を併用し、
// 絶対演算は assume_timezone → +Duration → to_timezone で「付け直す」手間がかかる
use time_tz::{PrimitiveDateTimeExt, OffsetDateTimeExt};
let start = naive.assume_timezone(ny).unwrap_first();
let later = (start + time::Duration::hours(24)).to_timezone(ny); // 03-09 02:30
```

jiffが最も意図を表しやすく、chronoがそれに次ぎ、timeは（IANA TZを扱う限り）手数が多いという差がそのまま出ます。

# まとめ

時刻は「デファクトが1つ」ではなく、しかも地殻変動の最中にある領域です。AIの提案を評価するときは、次のように整理すると実務的です。

- 既存資産と学習データ量を重視するなら **chrono**（ただしメンテ縮小の表明を踏まえ、adapterで隔離して「いつでも捨てられる」形にしておく）
- 依存最小、`no_std`、組込みなら **time**（IANAタイムゾーンを使うなら time-tz で依存が増える点に注意）
- タイムゾーン厳密、DST安全な演算、新規プロジェクトなら **jiff**（ただし1.0前であることと、silent failureを拾うためのテストをセットで）

そして全体を貫く教訓は、「型やメソッドで意味論を分けているか」と「AIが今のAPIで書けるか」を分けて見ることです。jiffはDST演算の意味論を明快に型で表す一方、`since` の向きのような同型の取り違えはすり抜けます。「`cargo build` が通った」だけで安心せず、時刻処理には値のテストを書くべきです。

次回（最終回）は GUI crate を扱います。GUIでは「ビルドが通る環境を作れるか」と「学習データがどこにあるか」が選定を支配する、という別の景色が見えてきます。
