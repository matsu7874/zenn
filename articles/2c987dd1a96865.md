前回（[AI時代のRust crate選定 (2) ３つのデータベース接続](#)）は、データベースアクセスを題材に「**コンパイル時検証はレビューの代替になる**」という観点を扱いました。AIが書いたSQLのスキーマ不整合を、コンパイラがレビュアーとして肩代わりしてくれる、という話です。

今回のテーマは **時刻** です。一見地味ですが、時刻処理は「それらしく書けているのに、実は間違っている」コードが生まれやすく、AIに書かせるうえで罠の多い領域です。そして選択肢は [chrono](https://crates.io/crates/chrono) / [time](https://crates.io/crates/time) / [jiff](https://crates.io/crates/jiff) の3つに割れています。

この記事の結論を先に言うと、時刻crateは次の2軸で見るべきだということです。

1. **タイムゾーンやDST（夏時間）の意味論を、型やメソッドで表現できるか**
2. **AIが「いまのAPI」で書けるか（学習データの量と新しさ）**

そして実際に3つのcrateへ書かせて検証すると、**一発書きの精度は学習データ量に素直（chrono > time > jiff）**でしたが、それ以上に **失敗の質が決定的に違う** ことが分かりました。先に要点だけ言うと、timeの失敗は**コンパイルエラーで安全に止まり**、jiffの失敗は**コンパイルを通り抜けて値だけ間違える**（silent failure）のです。

# 時刻処理でAIがやりがちな間違い

具体的なcrateの前に、時刻処理そのものの「踏みやすい罠」を押さえます。これは言語やcrateを問わず共通で、AIも人間も間違えるポイントです。

- **naive（タイムゾーンなし）と aware（タイムゾーンあり）の取り違え**: 「2026-06-15 12:00」という壁時計の文字列が、UTCなのかJSTなのか、情報が欠けたまま計算してしまう
- **「とりあえずローカル時刻」**: サーバーのタイムゾーン設定に依存し、本番とローカルで結果が変わる
- **DST（夏時間）を跨ぐ加減算**: 「1日後」と「24時間後」は同じではありません。夏時間の切り替え日には、壁時計上の「1日後」は物理的には23時間後や25時間後になります
- **月末の加算**: 「1月31日の1ヶ月後」は何日か。2月31日は存在しないので、2月28日（や29日）にクランプする必要があります

特に3つ目のDSTは直感に反します。例として、アメリカ東部（America/New_York）が夏時間に切り替わる **2026年3月8日** の午前1時30分を起点に考えます。

| 起点 | 演算 | 結果（壁時計） |
|---|---|---|
| 2026-03-08 01:30 (EST) | **暦の上で1日後**（カレンダー演算） | 2026-03-**09** 01:30 |
| 2026-03-08 01:30 (EST) | **物理的に24時間後**（絶対演算） | 2026-03-**09** **02:30** |

3月8日の午前2時に時計が1時間進むため、同じ「翌日」でも、壁時計を保つカレンダー演算では01:30、物理的な24時間後では02:30と、**1時間ずれます**。この区別をcrateがどう表現するかが、時刻crate選定の肝です。

# crateを評価する8つの観点

第2回と同じ8観点で評価します。「人間にとっての書きやすさ」ではなく「**AIに書かせたときに回りやすいか**」を見る観点でした。

| 観点 | チェック内容 | NG例 |
|---|---|---|
| AI相性 | AIが現在のAPIで動くコードを書ける | 存在しないメソッドや古いAPIを出す |
| 初速 | READMEから10分で動く | feature指定や別crate併用で詰まる |
| 復旧性 | エラーをAIに渡すと直る | 型やAPIが複雑すぎて直らない |
| 定番接続 | tokio/serde/[tracing](https://crates.io/crates/tracing)等と繋がる | 独自世界観が強すぎる |
| ビルド | 自分の環境で素直にbuildできる | C依存や重い依存で詰まる |
| 速度 | cargo check が待てる | 変更ごとに集中が切れる |
| 捨てやすさ | adapterで隔離できる | アプリ全体に染み込む |
| 保守 | 外部変化に追従している | 開発停止・非推奨化 |

時刻領域で特に効くのは次の3点です。

- **AI相性**: 時刻処理は「APIの新しさ」がそのまま精度に出ます。古くからあるcrateほどAIが書け、新しいcrateほど学習データが薄く外しやすい
- **保守**: この領域は「枯れている＝安全」が崩れました（後述。chronoは作者がメンテ縮小を表明）
- **ビルド・速度**: IANAタイムゾーン（`Asia/Tokyo`等）をどう持つかで依存とビルド時間が大きく変わる

# 時刻crateの三本柱

| crate | スタイル | タイムゾーンの扱い | 一言で言うと |
|---|---|---|---|
| chrono | 老舗・固定オフセット中心 | `DateTime<Tz>` / `NaiveDateTime`。IANAは [chrono-tz](https://crates.io/crates/chrono-tz) 併用 | 学習データ最多。事実上の旧デファクト |
| time | `no_std` 対応・依存最小 | `OffsetDateTime`（固定オフセット）。IANAは [time-tz](https://crates.io/crates/time-tz) 併用 | 組込み向き。featureが細分 |
| jiff | [Temporal](https://tc39.es/proposal-temporal/) 準拠・タイムゾーンファースト | `Zoned` がIANA TZを内蔵 | [ripgrep](https://crates.io/crates/ripgrep)作者による新標準候補（1.0手前） |

※バージョンは chrono 0.4.45 / time 0.3.49 / jiff 0.2.28（2026年6月時点）

## chrono: 老舗、ただし「ソフトdeprecated」

chrono はもっとも広く使われてきたcrateです。`DateTime<Utc>` や `NaiveDateTime` といった型はRust界隈で広く知られ、AIの学習データも豊富です。

```rust
use chrono::{Utc, TimeZone, Datelike, Timelike};
use chrono_tz::Asia::Tokyo;

// UTC epoch秒 → JST の壁時計
let jst = Utc.timestamp_opt(epoch, 0).single().unwrap().with_timezone(&Tokyo);
println!("{}:{}", jst.hour(), jst.minute());
```

ただし注意点が一つあります。2026年初頭、chronoの保守者が [Issue #1768](https://github.com/chronotope/chrono/issues/1768) で、APIデザインの古さと改修コストを理由に**今後メンテナンスを縮小し、後継として jiff を薦める**と表明しました。即座に危険になるわけではありませんが、「枯れていて安心」という前提が揺らいでいる点は、保守の観点で割り引く必要があります。

## time: 依存最小、`no_std` 対応

time は標準ライブラリとの相互運用を重視し、`no_std` でも使える軽量なcrateです。組込みや依存を絞りたい場面で選ばれます。

```rust
use time::OffsetDateTime;
use time::format_description::well_known::Rfc3339;

let odt = OffsetDateTime::parse("2026-06-15T12:00:00Z", &Rfc3339).unwrap();
let epoch = odt.unix_timestamp();
```

注意点は2つあります。1つは **featureフラグが細かく分かれている** こと（`formatting` / `parsing` / `macros` などが既定オフ）。もう1つは、**timeはIANAタイムゾーンを持たない** ことです。`Asia/Tokyo` のようなタイムゾーンを扱うには別crateの [time-tz](https://crates.io/crates/time-tz) を併用します。これが後述の通り、AIに書かせるうえでの躓きどころになります。

## jiff: Temporal準拠のタイムゾーンファースト

jiff は [BurntSushi](https://github.com/BurntSushi)（ripgrep / regex の作者）による新しいcrateで、JavaScriptの [Temporal](https://tc39.es/proposal-temporal/) 提案を下敷きにした設計です。最大の特徴は、**タイムゾーン付きの `Zoned` 型が標準で IANA タイムゾーンを扱える** こと、そして **カレンダー演算と絶対演算をメソッドで明確に分ける** ことです。

```rust
use jiff::{civil, ToSpan};

let z = civil::date(2026, 3, 8).at(1, 30, 0, 0).in_tz("America/New_York")?;
let cal = z.checked_add(1.day())?;    // 暦の1日後  => 2026-03-09 01:30
let abs = z.checked_add(24.hours())?; // 物理24時間後 => 2026-03-09 02:30
```

`1.day()`（カレンダー）と `24.hours()`（絶対）が別物として書き分けられ、DSTを跨ぐと結果が変わる——という意味論がコードに表れます。本記事の実測でも、この区別は**全試行が正しく扱えました**。一方で、jiff は **まだ 1.0 未到達**（0.2系で破壊的変更の余地あり）で、学習データも薄く、後述の通り別の落とし穴があります。

# 8つの観点で三本柱を評価する

実際に Claude Opus 4.8 へ各crateで同じ8課題を書かせ、ビルドと値の検証で採点した結果（下の実測メモ）を反映した評価が次の表です。◎○△は「AIに書かせる前提での相対評価」です。

| 観点 | chrono | time | jiff |
|---|---|---|---|
| AI相性 | ◎ 一発書き 5/5。学習データ最多 | ○ 3/5。time-tzのトレイト作法で躓く | △ 2/5。新しいAPIをsilentに誤る |
| 初速 | ○ つなぐだけは素直 | △ feature細分＋time-tz併用 | ◎ タイムゾーン込みで素直 |
| 復旧性 | ◎ そもそも落ちない | ◎ コンパイラが具体的に弾く | ○ 直るが**テストが要る**（型では気づけない） |
| 定番接続 | ◎ serde等に広く対応 | ○ serde対応、TZは別crate | ◎ serde/RFC9557、jiff-sqlx等 |
| ビルド | ○ ほぼPure Rust | △ time-tzで依存が15に増える | ◎ 依存2・最速・Pure Rust |
| 速度 | ○ `cargo check` 2.97s | △ 4.63s | ◎ 0.93s |
| 捨てやすさ | ○ | ○ | ○ |
| 保守 | △ 作者がメンテ縮小・jiff推奨を表明 | ◎ 活発 | ○ 活発だが **1.0未達** |

第2回ではDB3crateとも保守は◎でしたが、今回は **「保守」で初めて△が出る** のが見どころです。「枯れている＝安全」が成立しない領域だということです。

以下、特に判断を左右する観点を掘り下げます。

## AI相性・復旧性 — 一発書きの精度と「失敗の質」

学習データ量の仮説（chrono最多）どおり、一発書きの精度は **chrono 5/5 > time 3/5 > jiff 2/5** でした。しかし重要なのは精度そのものより、**失敗の質**です。

- **chrono**: `dt - dt`、`+ Duration`、`checked_add_months` など直感的なAPIで、5/5。失敗なし
- **time**: 失敗2件は **どちらもコンパイルエラー**。`Asia/Tokyo` への変換に time-tz の拡張トレイト（`use time_tz::OffsetDateTimeExt;`）が必要なのに import を落とし、`to_timezone` がメソッド解決できず弾かれた。**間違った値は出ない（安全側）**
- **jiff**: 失敗3件は **コンパイルが通ったのに値を間違えた**。「2つの日付の差」を求める課題で、新しめのAPI `Date::since` の差の向きを誤り、**+28日であるべきところを -28日（符号が逆）** で返した。3試行とも同じ間違いです

ここが核心です。第2回の「コンパイル時検証がレビューを肩代わりする」は、**DST/タイムゾーンの意味論（カレンダーか絶対か）には効きます**——jiffの `1.day()` / `24.hours()` のように、メソッドや型で意味が分かれていれば、取り違えはコンパイルや結果に表れます。しかし、**`since` の向きのように「同じ型（`i64`）で符号だけ違う」取り違えは、型では防げません**。テストを書いて値を検証しない限り、`cargo build` が通った時点では気づけないのです。

:::message
**実測メモ**（Claude Opus 4.8 / 各crate5試行 / ドキュメント参照を禁じた「一発書き」/ 8課題をビルドと値で判定）

- 一発書き合格: **chrono 5/5・time 3/5・jiff 2/5**
- 失敗の質: time（2件）は**コンパイルエラー**（time-tzのトレイト未import等）。jiff（3件）は**silent failure**（`Date::since` の差の符号を誤り -28 を返す。型は合うので通る）
- 修正ループ: 失敗5件すべて、エラー出力（timeはコンパイルエラー / jiffはテストの値不一致）だけを返したら **1ラウンドで全て修正**できた。ただし jiff は**テストが無いと失敗に気づけない**点が time と異なる
:::

つまり jiff は「一発書きの精度が最も低く、しかも失敗がコンパイルをすり抜ける」ため、**使うならテストとセットが前提**になります。逆に言えば、テスト体制があれば、jiffの意味論の明快さ（DST安全な演算）と軽さは大きな武器です。

## 保守 — 「枯れている」が逆転した

通常、ライブラリ選定では「枯れていて広く使われている」ことが安心材料です。chronoはまさにそれでした。しかし2026年、chronoの保守者自身が [#1768](https://github.com/chronotope/chrono/issues/1768) でメンテ縮小と jiff 推奨を表明したことで、この前提が崩れました。

とはいえ、chronoが即座に動かなくなるわけではありません。膨大な既存資産があり、AIの学習データも最多です。一方の後継候補 jiff は **まだ 1.0 に達しておらず**、破壊的変更の余地があります。つまり「老舗だが先細り（chrono）」対「将来有望だが未成熟（jiff）」という、どちらも単純に手放しで薦めにくい構図です。この観点こそ、AIの提案を鵜呑みにせず人間が確認する価値があります。

## ビルド・速度・featureの罠

依存とビルド時間の実測値です（クリーンビルド、同一環境）。

| crate | 依存数（ユニーク） | C依存 | release ビルド | `cargo check` |
|---|---|---|---|---|
| jiff | **2** | なし（Pure Rust） | **1.89s** | **0.93s** |
| chrono | 10 | core-foundation-sys（OS標準。追加インストール不要） | 3.35s | 2.97s |
| time（単体） | 7 | なし | — | — |
| time + time-tz | **15** | libc | 5.98s | 4.63s |

jiffは依存2・最速で、IANAタイムゾーン込みでこれです。chronoは中庸。timeは**単体なら7と最小ですが、IANAタイムゾーンのために time-tz を足すと15へ倍増**し、ビルドも最長になります（`cargo check` 4.63秒は、修正ループ1回あたりの単価として効いてきます）。

なお、いずれも第2回のdieselの `libpq` のような**外部システムライブラリのインストールは不要**です。この領域に「C依存でビルドが詰まる」壁はありません。

加えて time は **featureが細分** されており、`OffsetDateTime::parse` には `parsing`、整形には `formatting`、`format_description!` マクロには `macros` を、それぞれ明示的に有効化する必要があります。最小featureから始めると、AIが整形・パースを書いた瞬間にコンパイルエラーで足踏みします。

# 実例: 同じ「DST安全な加算」を3 crateで

最後に、本記事の山場である「カレンダー演算と絶対演算の区別」を、3つのcrateでどう書くか並べます。

```rust
// jiff: メソッドで意味論が分かれる（最も明快）
let z = jiff::civil::date(2026, 3, 8).at(1, 30, 0, 0).in_tz("America/New_York")?;
z.checked_add(1.day())?;    // 暦の1日後   => 03-09 01:30（壁時計を保つ）
z.checked_add(24.hours())?; // 物理24時間後 => 03-09 02:30（DSTで1時間ずれる）
```

```rust
// chrono: Days（カレンダー）と Duration（絶対）で区別。chrono-tz が必要
use chrono::{Days, Duration, TimeZone};
use chrono_tz::America::New_York;
let dt = New_York.from_local_datetime(&naive).single().unwrap();
dt + Days::new(1);        // カレンダー => 03-09 01:30
dt + Duration::hours(24); // 絶対     => 03-09 02:30
```

```rust
// time: IANA TZ を持たないので time-tz を併用し、
// 絶対演算は assume_timezone → +Duration → to_timezone で「付け直す」手間がかかる
use time_tz::{PrimitiveDateTimeExt, OffsetDateTimeExt};
let start = naive.assume_timezone(ny).unwrap_first();
let later = (start + time::Duration::hours(24)).to_timezone(ny); // 03-09 02:30
```

jiffが最も意図を表しやすく、chronoがそれに次ぎ、timeは（IANA TZを扱う限り）手数が多い、という差がそのまま出ます。

# まとめ: 枯れてもいない、新しすぎてもいい、ではない

時刻は「デファクトが1つ」ではなく、しかも地殻変動の最中にある領域です。AIの提案を評価するときは、次のように整理すると実務的です。

- **既存資産・学習データ量を重視**するなら **chrono**（ただしメンテ縮小の表明を踏まえ、長期では移行も視野に）
- **依存最小・`no_std`・組込み**なら **time**（IANAタイムゾーンが要るなら time-tz で依存が増える点に注意）
- **タイムゾーン厳密・DST安全な演算・新規プロジェクト**なら **jiff**（ただし1.0前であることと、**silent failureを拾うためのテスト**をセットで）

そして全体を貫く教訓は、**「型やメソッドで意味論を分けているか」と「AIが今のAPIで書けるか」を分けて見る** ことです。jiffはDST演算の意味論を明快に型で表す一方、`since` の向きのような同型の取り違えはすり抜けます。「`cargo build` が通った」だけで安心せず、時刻処理には値のテストを——これが今回の要点です。

次回（最終回）は **GUI crate** を扱います。GUIでは「ビルドが通る環境を作れるか」と「学習データがどこにあるか」が選定を支配する、という別の景色が見えてきます。
