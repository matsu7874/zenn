SWEの松本([@matsu7874](https://twitter.com/matsu7874))です。前回（[AI時代のRust crate選定 (1) デファクトスタンダードなcrate群](https://zenn.dev/estie/articles/4e057582208b0b)）は、[serde](https://crates.io/crates/serde) や [tokio](https://crates.io/crates/tokio) のようにデファクトスタンダードなcrateがある領域を紹介しました。デファクトを外す提案をAIがしてきたら理由を聞くべき、という整理でした。

Rustの全ての領域にデファクトスタンダードがあるわけではありません。今回扱うWebバックエンドとデータベースアクセスは、有力な少数の選択肢から選ぶ領域です。特にデータベースアクセスは [sqlx](https://crates.io/crates/sqlx) / [diesel](https://crates.io/crates/diesel) / [sea-orm](https://crates.io/crates/sea-orm) の3つに割れています。

結論から書くと、迷ったら sqlx です。`query!` マクロがSQLを実在のDBスキーマとコンパイル時に照合するため、AIが書いたSQLの列名・型・テーブル名の不整合を、人間が目視でレビューする前にコンパイラが弾いてくれます。

Webバックエンド側は axum を起点に考える前提だけ確認し、この記事の主眼はデータベースアクセスcrateの比較に置きます。

# crateを評価する4つの観点

機能が足りているかはAIに判断させる前提で、人間が見るべきはAIに書かせたときに実装ループが回るかです。観点を「チェック内容」と「NG例」で4つに整理しました。

| 観点 | チェック内容 | NG例 |
|---|---|---|
| AI親和性 | AIが現行APIでコードを書け、エラーを渡せば修正できる | 古い/存在しないAPIを出し、エラーを見ても修正できない |
| 立ち上げと反復速度 | READMEから環境構築が簡単で、build と `cargo check` の待ちが短い | feature指定やC依存で詰まり、修正のたびに待たされる |
| エコシステム適合 | tokio/serde/tracing等と自然に繋がる | 独自世界観が強すぎる |
| 撤退可能性 | adapterで隔離でき、メンテが止まっても捨てられる | アプリ全体に広がり、放置されても逃げられない |

4観点とも、見ているのはAIに書かせたときのループの回りやすさです。背景を補足します。

- **AI親和性**: AIエージェントは「書く→コンパイル→修正」ループを1タスクで何十回も回す。一発目が多少外れても、エラーが具体的で修正できるなら実用上の損失は小さい。逆に、エラーが難解で修正できないとエージェントが詰まってしまう。普段馴染みのない領域を扱うときほど、AIが使えるライブラリを選ぶ価値が大きくなる。
- **立ち上げと反復速度**: READMEからシンプルに動くか、`cargo build` / `cargo check` の待ちが短いかは「編集→確認」ループごとのコストである。特にC依存のビルドエラー（「OpenSSL/libpqが見つからない」）はAIエージェントの環境構築失敗の最頻出パターンである。Pure Rustで完結する構成なら `cargo build` が通る環境さえあればよく、環境構築ごとAIに任せられる。コンパイル時間も、修正ループ1回あたりのコストとして数十回ぶん積み上がる。
- **エコシステム適合**: 周辺crateと自然に繋がるかを見る。Webバックエンドなら tokio / [tower](https://crates.io/crates/tower) / serde / [tracing](https://crates.io/crates/tracing) / [rustls](https://crates.io/crates/rustls) などと素直に組み合わせられるほど、AIが既存のサンプルやエラーから修正しやすくなる。逆に独自の抽象やランタイムに寄るほど、周辺知識の翻訳コストが増える。
- **撤退可能性**: AIに任せると依存はどんどん増える。それでも怖くないのは、adapter（外部crateとの境界に置く変換層）で隔離して、ダメになったら捨てられるからである。馴染みのない領域のvibe codingでは、crateを育てる側ではなく利用者側に回ることが多いはずである。だからこそ「保守が完璧か」より「捨てやすいか」を重視する。メンテが細っても、隔離してあれば差し替えられる。

# Webフレームワークは axum で考える

まずWebフレームワーク側です。現在のRust製Webバックエンドでは、tokioエコシステムに素直に乗るという観点で [axum](https://crates.io/crates/axum) が最も無難な起点です。

歴史的には [actix-web](https://crates.io/crates/actix-web) が高いパフォーマンスで人気を集めました。actix-web が不適切という話ではありませんが、いま新規にAIへ書かせる前提なら axum を起点に考えるのが無難です。

- 非同期ランタイムは tokio 一強（前回参照）。axum はその tokio チームが開発しており、tokio / [tower](https://crates.io/crates/tower) / [hyper](https://crates.io/crates/hyper) という同一エコシステムの上に乗っている
- ミドルウェアは tower の `Service` トレイトとして共通化されており、タイムアウトやリトライ、レート制限などを使い回せる
- 公式や準公式のサンプルが豊富で、LLMが正確に書きやすい

```toml
[dependencies]
axum = "0.8"
tokio = { version = "1", features = ["full"] }
```

axum 側はこの記事の主役ではないので、「tokioエコシステムにシンプルに乗りたいなら axum」とだけ押さえて、本題のデータベースに進みます。

# データベースアクセスの3つの選択肢

Rustでデータベース（主にPostgreSQL / MySQL / SQLite）にアクセスする場合、有力な選択肢は次の3つです。それぞれ設計思想が大きく異なります。

| crate | スタイル | 検証タイミング | 特徴 |
|---|---|---|---|
| sqlx | 生SQL + マクロ | コンパイル時 | SQLをそのまま書き、実在DBに照合して型を検証 |
| diesel | 型安全DSL | コンパイル時 | SQLをRustのDSLで組み立て、型で検証 |
| sea-orm | ORM (ActiveRecord的) | Rust側の型 + ランタイム | エンティティ操作中心。動的なクエリ構築が得意 |

## sqlx: 生SQLをコンパイル時に検証する

sqlx は「SQLはSQLとして書く」というスタイルです。`query!` マクロにSQL文字列を渡すと、ビルド時に実際のデータベース（または保存済みスキーマ情報）へ接続してクエリを検証し、結果の列名や型からRustの構造体を生成します。

```rust
// コンパイル時に DB スキーマと照合される
let user = sqlx::query!(
    "SELECT id, name FROM users WHERE id = $1",
    user_id
)
.fetch_one(&pool)
.await?;

// user.id, user.name は型付きでアクセスできる
println!("{}: {}", user.id, user.name);
```

テーブル名や列名を間違えたり、型が合わなかったりすれば、実行する前にコンパイルが失敗します。SQL構文そのものは慣れた書き方が使え、なおかつ静的に守られます。

ただし、この恩恵は `query!` / `query_as!` のように、コンパイル時にSQL文字列を見られるマクロが中心です。リクエスト条件に応じてSQL文字列を組み立てる動的クエリや `QueryBuilder` では、同じ粒度のコンパイル時検証は効きません。その場合は、実DBを使ったrepository単位のテストで補う必要があります。

## diesel: SQLをRustのDSLで組み立てる

diesel は生SQLを書かず、Rustのコードとしてクエリを組み立てます。スキーマ定義から型を生成し、クエリビルダがすべて型レベルで整合性を保証します。

```rust
use crate::schema::users::dsl::*;

let result = users
    .filter(id.eq(user_id))
    .select((id, name))
    .first::<(i32, String)>(&mut conn)?;
```

こちらもコンパイル時に安全ですが、検証の方法が違います。sqlx が「実在DBにSQLを照合する」のに対し、diesel は「RustのDSLとして型が通るか」で守ります。SQLの知識をdiesel独自のDSLに翻訳して書く必要があり、ここがAIに書かせるうえでの分かれ目になります（後述）。

なお diesel は同期APIが基本で、非同期に乗せるには [diesel-async](https://crates.io/crates/diesel-async) を併用します。

diesel のコンパイル時検証が見ているのは、DBそのものではなく、`diesel print-schema` などで生成されたRust側のスキーマ定義です。そのため、migration と `schema.rs` がずれていれば、コンパイルが通っても実DBとは合わない可能性があります。CIの運用は、migration 適用、スキーマ生成、`cargo check` を同じ流れで回す前提を置くべきです。

## sea-orm: ORMとして動的に組み立てる

sea-orm はいわゆるORMで、他言語のActiveRecordに近い感覚です。エンティティを定義し、それに対するメソッドでデータを操作します。非同期前提で、sqlx の上に構築されています。

```rust
let user: Option<user::Model> = User::find_by_id(user_id).one(&db).await?;
```

実行時に条件を組み立てる動的なクエリ（検索条件がリクエスト次第で変わる管理画面など）が書きやすい一方、sqlx の `query!` のように実在DBへ照合してSQL文字列をコンパイル時検証する仕組みとは性質が違います。生成されたエンティティやカラム名の型で守られる範囲はありますが、最終的なクエリの意味やDBスキーマとのずれは、テスト実行まで残りやすくなります。

これは sea-orm が安全でないという意味ではありません。エンティティ生成、migration、型付きのカラム指定で防げるミスはあります。ただ、AIが書いたSQLをコンパイル時に実DBへ照合するという効果は、sqlx の `query!` ほど直接的ではない、という整理です。

# 4つの観点で評価する

先ほどの4観点を sqlx / diesel / sea-orm に当てたのが下表です。実際に Claude Opus 4.8 へ書かせて検証した結果（後述の実測）を反映しています。

| 観点 | sqlx | diesel | sea-orm |
|---|---|---|---|
| AI親和性 | 単純SQLは一発で書ける。集計/JOINはマクロ作法に注意 | DSLへの翻訳が要るが、外しても型が必ず弾く | API作法で失敗しやすいが、修正は容易 |
| 立ち上げと反復速度 | PostgreSQL / MySQL はPure Rust。`query!` の検証にDB接続かオフラインキャッシュが要る | libpq等のC依存＋schema生成。型レベル計算でビルドも重い | 繋ぐだけならシンプル。マクロ＋sqlxでビルドはやや重い |
| エコシステム適合 | tokio/axum/rustlsにそのまま乗る | 独自寄り（非同期は diesel-async） | sqlx上に構築 |
| 撤退可能性 | 生SQLで隔離しやすい | 型がアプリに広がり差し替えづらい | エンティティが広がりやすい |

※3つとも開発は活発で、「いま保守が止まる」心配は小さいです。

以下、特に判断を左右した観点を順に見ます。

## 実測：一発書きの成績と修正ループ

Claude Opus 4.8 に、ドキュメント参照を禁じた「一発書き」で各5試行、スキーマは生成済みとしてクエリ本体のみ書かせ、実DBでテストしました。著者環境での小さな検証で、汎用ベンチマークではありません。

- 単純なブログCRUD: diesel 5/5・sqlx 5/5・sea-orm 2/5
- 多対多・自己参照・LEFT JOIN集計を含む複雑モデル: diesel 3/5・sea-orm 2/5・sqlx 1/5

生SQLは言語を超えた共通知識なので sqlx が有利、と予想していました。単純なCRUDではその通りでしたが、複雑モデルでは sqlx が最下位です。失敗の内訳は次の通りです。

- **sqlx**: 5試行中4回が `query!` マクロ固有の作法（`AS "col!: T"` の型やNULL注釈、その別名を `ORDER BY` で参照できない等）で失敗しました。SQL自体は正しいのにマクロで落ちるわけです
- **diesel**: SQLを独自DSLに翻訳する必要はありますが、AIはidiomaticに書けることが多いです。外部結合の `nullable()` など型の作法を外すと、型不整合として検出されやすいです
- **sea-orm**: ORMの発想は他言語から類推が効きます。躓きどころは `PaginatorTrait` の import や join 補助メソッドの選択といったAPI表面の作法です

「diesel は独自DSLだから精度が落ちる」という単純な順位付けは観測されませんでした。

ただし今回の失敗は、いずれもコンパイルエラーまたは実DBテストの失敗として検出できました。エラーを返すと1〜2ラウンドで全て修正できています。一発書きの精度そのものより、外したときにエラーから復旧できるかで評価するべきだと分かります。

## 立ち上げと反復速度では diesel がCライブラリを引きやすい

「OpenSSL/libpqが見つからない」というC依存のビルドエラーは、AIエージェントの環境構築失敗の最頻出パターンです。DBアクセスでも同じ構図が出ます。

- **sqlx**: PostgreSQL / MySQL ドライバはPure Rustで、libpq を必要としません。TLSも rustls を選べば `cargo build` が通る環境だけで完結します。SQLite は `libsqlite3-sys` を使うため、ここでは主にPostgreSQL / MySQL接続の話として読んでください
- **diesel**: PostgreSQL バックエンドは既定で libpq（`pq-sys`）、MySQL は libmysqlclient といったC依存を引きやすい構成です。システムにライブラリや pkg-config が要求され、Dockerイメージやクロスコンパイル（musl静的リンク、ARM）が一段面倒になります
- **sea-orm**: 土台が sqlx なので、こちらもPure Rustで完結させられます

ここでいうPure Rustは、PostgreSQL / MySQL で rustls などのRust実装を選ぶ構成の話です。TLSに native-tls を選べばOpenSSL等を引くことがありますし、SQLiteでは `libsqlite3-sys` を意識する場面があります。feature選択で依存関係が変わる点は、AIに任せる場合ほど明示しておく必要があります。

実際に libpq / pg_config の無い環境で PostgreSQL バックエンドをビルドすると、diesel は `ld: library 'pq' not found` でリンクに失敗し、sqlx / sea-orm はそのまま通りました。diesel の型安全DSL自体は成熟しているので、環境を整えられるチームでは強い選択肢ですが、AIに環境構築ごと任せる前提では分が悪いです。

初回の段取りにも差が出ます。つなぐだけなら sea-orm がシンプル。sqlx は `query!` の検証にビルド時のDB接続か `cargo sqlx prepare` のオフラインキャッシュが要り、diesel は `diesel print-schema` でのスキーマ生成とC依存の用意が要ります。反復速度（`cargo check`）も diesel / sea-orm は型レベル計算や proc-macro で伸びがちで、修正ループ1回あたりのコストとして積み上がります。

## エコシステム適合と撤退可能性

- **エコシステム適合**: sqlx は tokio / axum / rustls にそのまま乗り、sea-orm はその sqlx の上に構築されています。diesel は独自エコシステムの色が濃く、非同期化には diesel-async、接続プールには [r2d2](https://crates.io/crates/r2d2) / [deadpool](https://crates.io/crates/deadpool) など、別部品の組み合わせが要ります
- **撤退可能性**: sqlx は生SQL文字列なので repository / adapter で隔離しやすく、ダメになっても差し替えやすいです。diesel は `Queryable` / `table!` の型が、sea-orm はエンティティがアプリ側に広がりやすく、後から剥がしづらくなります

## コンパイル時検証はレビューの一部を代わってくれる

AIが生成したコードで一番こわいのは、それらしく書けているSQLが実は列名を間違えている、型が合っていない、といった類のミスです。これを人間が毎回目視レビューだけで拾うのは難しいです。

sqlx と diesel は、この種のミスをコンパイルエラーとして弾いてくれます。

- 存在しない列を参照した → コンパイルが通らない
- `i32` を期待する場所に `String` を渡した → コンパイルが通らない
- テーブル名のタイポ → コンパイルが通らない

AIが書いたSQLの「スキーマとの整合性」という、人間レビューでもっとも見落としやすい部分を、コンパイラがレビュアーとして代わりに確認してくれるわけです。AIに「書く→コンパイル→修正」を回させておけば、列名・型・テーブル名の不整合はコミット前に弾けます。

正確には、sqlx が照合するのはビルド時に接続したDB、または `.sqlx` のオフラインキャッシュです。diesel が照合するのは生成済みのRust側スキーマです。どちらも migration と検証対象がずれていると意味が薄くなるため、CIで「migrationを適用した状態を検証対象にする」流れを作る必要があります。

とはいえ、コンパイル時検証が確認してくれるのはスキーマ整合の範囲だけです。WHERE条件の意味、JOINの向き、N+1、権限条件の抜け、インデックス設計などは別途レビューやテストが必要です。それでも、AIに任せる範囲が広いほど「コンパイルが通った＝スキーマ整合の大部分は確認済み」という保証の価値は大きくなります。

# 推奨は sqlx を起点にすること

以上を踏まえた推奨は sqlx です。

1. 単純なクエリなら生SQLそのままで一発目の精度が高く、外しても具体的なエラーで修正できる（集計やJOINでは `query!` マクロの作法に一手間かかる点は割り引く）
2. `query!` マクロが実在スキーマと照合し、コンパイル時検証でスキーマ整合性レビューの一部を代わってくれる
3. axum / tokio と同じ非同期エコシステムにシンプルに乗り、PostgreSQL / MySQL ではPure Rustで `cargo build` が通る

複雑なクエリの一発書きでは diesel に負けるケースがあっても、初速・環境構築・撤退可能性まで含めた実務の既定値として sqlx を起点に置く、という重み付けです。

他の2つが向く場面もあります。

- **diesel が向く**: 生SQLを一切書きたくない、Rust側の型で完全に閉じたいチーム
- **sea-orm が向く**: 検索条件が動的に変わる管理画面やCRUD中心の業務アプリで、クエリを動的に組み立てたい場合

「迷ったら sqlx、動的クエリ主体なら sea-orm、DSLで閉じたいなら diesel」という整理が実務的です。

# axum + sqlx の最小構成

最後に、axum と sqlx を組み合わせた最小のAPIを示します。`query!` マクロがコンパイル時にスキーマと照合される様子が見どころです。

```toml
[dependencies]
axum = "0.8"
tokio = { version = "1", features = ["full"] }
sqlx = { version = "0.8", features = ["runtime-tokio", "postgres"] }
serde = { version = "1", features = ["derive"] }
anyhow = "1"
```

```rust
use axum::{extract::{Path, State}, routing::get, Json, Router};
use serde::Serialize;
use sqlx::PgPool;

#[derive(Serialize)]
struct User {
    id: i32,
    name: String,
}

async fn get_user(
    State(pool): State<PgPool>,
    Path(user_id): Path<i32>,
) -> Result<Json<User>, axum::http::StatusCode> {
    // この query! は **ビルド時に** DB スキーマと照合される。
    // 列名や型を間違えれば、ここでコンパイルが失敗する。
    let rec = sqlx::query!(
        "SELECT id, name FROM users WHERE id = $1",
        user_id
    )
    .fetch_one(&pool)
    .await
    .map_err(|_| axum::http::StatusCode::NOT_FOUND)?;

    Ok(Json(User { id: rec.id, name: rec.name }))
}

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let pool = PgPool::connect(&std::env::var("DATABASE_URL")?).await?;

    let app = Router::new()
        .route("/users/{id}", get(get_user))
        .with_state(pool);

    let listener = tokio::net::TcpListener::bind("0.0.0.0:3000").await?;
    axum::serve(listener, app).await?;
    Ok(())
}
```

`users` テーブルに `name` 列が無ければ、このコードはビルドに失敗します。AIがこのコードを書いて `cargo build` を回した時点で、スキーマ不整合は自動的に弾かれます。

※`query!` のコンパイル時検証には、ビルド時にDBへ接続できる環境か、`cargo sqlx prepare` で生成した `.sqlx` ディレクトリ（オフライン用のスキーマキャッシュ）が必要です。CIやAIエージェントの環境では後者を使うのが定石です。ただし `.sqlx` が古いままだと古いスキーマに対して検証してしまうため、migration 変更時にキャッシュを更新する運用までセットで必要です。

# まとめ

以上、Webバックエンドは axum を、データベースアクセスは sqlx を起点に考えるという整理と、その根拠になる4観点での比較を紹介しました。一発書きの精度は「crateの優劣」より「クエリの複雑さ × crate固有の作法」で決まること、それでもコンパイル時検証がスキーマ不整合をコミット前に弾いてくれることが今回の要点です。

次回は時刻crate（[chrono](https://crates.io/crates/chrono) / [time](https://crates.io/crates/time) / [jiff](https://crates.io/crates/jiff)）を扱います。
