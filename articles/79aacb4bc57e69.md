前回（[AI時代のRust crate選定 (1) デファクトスタンダードなcrate群](#)）の記事では、[serde](https://crates.io/crates/serde) や [tokio](https://crates.io/crates/tokio) のようにデファクトスタンダードなcrateがある領域を紹介しました。この領域でAIが別の提案をしてきたら特別の意図を確認するべきと整理しました。

Rustの全ての領域でデファクトスタンダードなライブラリがあるわけではありません。今回扱う **Webバックエンドとデータベースアクセス** は、有力な少数の選択肢からの選択が必要な領域です。
特にデータベースアクセスは [sqlx](https://crates.io/crates/sqlx) / [diesel](https://crates.io/crates/diesel) / [sea-orm](https://crates.io/crates/sea-orm) と有力な選択肢が３つあります。

この記事の結論を先に言うと、**コンパイル時検証はレビューの代替になる** ということです。AIが書いたSQLを人間が一行ずつ目視レビューする代わりに、コンパイラにレビュアーをやらせる。これが今回のキーメッセージです。

# crateを評価する8つの観点

機能が足りているかはAIに判断させる前提で、人間が見るべきは「**AIに書かせたときに回りやすいか**」です。観点を「チェック内容」と「NG例」で8つに整理しました。

| 観点 | チェック内容 | NG例 |
|---|---|---|
| AI相性 | AIが現在のAPIで動くコードを書ける | 存在しないメソッドを頻繁に出す |
| 初速 | READMEから10分で動く | feature指定や環境構築で詰まる |
| 復旧性 | エラーをAIに渡すと直る | 型やmacroが複雑すぎて直らない |
| 定番接続 | tokio/serde/[tracing](https://crates.io/crates/tracing)等と繋がる | 独自世界観が強すぎる |
| ビルド | 自分の環境で素直にbuildできる | openssl-sys やC依存で詰まる |
| 速度 | cargo check が待てる | 変更ごとに集中が切れる |
| 捨てやすさ | adapterで隔離できる | アプリ全体に染み込む |
| 保守 | 外部変化に追従している | OAuth/TLS/API clientが放置されている |

どれも「人間にとっての書きやすさ」ではなく「**AIに書かせたときの回りやすさ**」を見ているのがポイントです。特に効いてくる3点を補足します。

- **AI相性・復旧性が生産性の本体**: AIエージェントは「書く→コンパイル→直す」ループを1タスクで何十回も回します。一発目が多少外れても、**エラーが具体的で直せる**なら実用上の損失は小さい。逆に、一発で書けてもエラーが難解で直らないと一気に詰まります
- **速度はループ1回あたりの単価**: コンパイル時間は待ち時間ではなく、修正ループ1回あたりの単価として数十倍に増幅されます。`cargo check` の体感が悪いと集中が切れます
- **ビルドはAIの最頻出のつまずき**: C依存のビルドエラー(「OpenSSL/libpqが見つからない」)はAIエージェントの環境構築失敗の最頻出パターンです。Pure Rustなら `cargo build` が通る環境さえあればよく、**環境構築ごとAIに任せられます**


# Webフレームワークは axum で考える

まずWebフレームワーク側から。前回 [axum](https://crates.io/crates/axum) を「0.8のまま広く本番利用されている」例として挙げました。現在のRust製Webバックエンドでは、この axum がデファクトに最も近い位置にいます。

歴史的には [actix-web](https://crates.io/crates/actix-web) が高いパフォーマンスで人気を集めましたが、いま新規にAIへ書かせる前提なら axum を起点に考えるのが無難です。理由は「**定番接続**」（tokioエコシステムにそのまま乗れるか）に集約されます。

- 非同期ランタイムは tokio 一択（観点: 前回参照）。axum はその tokio チームが開発しており、tokio / [tower](https://crates.io/crates/tower) / [hyper](https://crates.io/crates/hyper) という同一エコシステムの上に乗っている
- ミドルウェアは tower の `Service` トレイトとして共通化されており、タイムアウト・リトライ・レート制限などを使い回せる
- 公式・準公式のサンプルが豊富で、「**AI相性**」の観点でもLLMが正確に書きやすい

```toml
[dependencies]
axum = "0.8"
tokio = { version = "1", features = ["full"] }
```

axum 側はこの記事の主役ではないので、ここでは「tokioエコシステムに素直に乗りたいなら axum」とだけ押さえて、本題のデータベースに進みます。

# データベースアクセスの三つ巴

Rustでデータベース（主にPostgreSQL / MySQL / SQLite）にアクセスする場合、有力な選択肢は次の3つです。設計思想がはっきり異なります。

| crate | スタイル | 検証タイミング | 一言で言うと |
|---|---|---|---|
| sqlx | 生SQL + マクロ | **コンパイル時** | SQLをそのまま書き、実在DBに照合して型を検証 |
| diesel | 型安全DSL | **コンパイル時** | SQLをRustのDSLで組み立て、型で検証 |
| sea-orm | ORM (ActiveRecord的) | ランタイム | エンティティ操作中心。動的なクエリ構築が得意 |

## sqlx: 生SQLをコンパイル時に検証する

sqlx は「SQLはSQLとして書く」というスタイルです。`query!` マクロにSQL文字列を渡すと、**ビルド時に実際のデータベース（または保存済みスキーマ情報）へ接続してクエリを検証** し、結果の列名・型からRustの構造体を生成します。

```rust
// コンパイル時に DB スキーマと照合される
let user = sqlx::query!(
    "SELECT id, name FROM users WHERE id = $1",
    user_id
)
.fetch_one(&pool)
.await?;

// user.id, user.name は型付きでアクセスできる
println!("{}: {}", user.id, user.name);
```

もしテーブル名や列名を間違えたり、型が合わなかったりすれば、**実行する前にコンパイルが失敗** します。SQL構文そのものは慣れた書き方が使え、なおかつ静的に守られるのが特徴です。

## diesel: SQLをRustのDSLで組み立てる

diesel は生SQLを書かず、Rustのコードとしてクエリを組み立てます。スキーマ定義から型を生成し、クエリビルダがすべて型レベルで整合性を保証します。

```rust
use crate::schema::users::dsl::*;

let result = users
    .filter(id.eq(user_id))
    .select((id, name))
    .first::<(i32, String)>(&mut conn)?;
```

こちらもコンパイル時に安全ですが、検証の方法が違います。sqlx が「実在DBにSQLを照合する」のに対し、diesel は「RustのDSLとして型が通るか」で守ります。SQLの知識をdiesel独自のDSLに翻訳して書く必要があり、ここがAIに書かせるうえでの分かれ目になります（後述）。

なお diesel は同期APIが基本で、非同期に乗せるには [`diesel-async`](https://crates.io/crates/diesel-async) を併用します。

## sea-orm: ORM として動的に組み立てる

sea-orm はいわゆるORMで、他言語のActiveRecordに近い感覚です。エンティティを定義し、それに対するメソッドでデータを操作します。非同期前提で、sqlx の上に構築されています。

```rust
let user: Option<user::Model> = User::find_by_id(user_id).one(&db).await?;
```

実行時に条件を組み立てる動的なクエリ（検索条件がリクエスト次第で変わる管理画面など）が書きやすい一方、**クエリの正しさはランタイムまで分からない** のが大きな違いです。

# 8つの観点で三つ巴を評価する

先ほどのチェックリストを sqlx / diesel / sea-orm に当てたのが下表です。◎○△は「AIに書かせる前提での相対評価」だと思ってください。

| 観点 | sqlx | diesel | sea-orm |
|---|---|---|---|
| AI相性 | ○ 単純SQLは◎、集計/JOINでマクロ作法に躓く | ○ idiomaticに書ける。外しても型で弾く | ○ import/API作法で躓くことがある |
| 初速 | △ マクロ検証にDB接続 or オフラインキャッシュが要る | △ schema生成＋C依存の用意が要る | ○ つなぐだけなら素直 |
| 復旧性 | ◎ エラーが具体的で直しやすい | ○ 型エラーは難解だが必ず出る | ○ import不足等は直しやすい |
| 定番接続 | ◎ tokio/axum/[rustls](https://crates.io/crates/rustls)にそのまま | △ 独自寄り（非同期は diesel-async） | ○ sqlx上に構築 |
| ビルド | ◎ Pure Rust | △ libpq 等のC依存を引きやすい | ◎ sqlx土台でPure Rust |
| 速度 | ○ マクロ分だけ増える | △ 型レベル計算で重い | △ マクロ＋sqlxで重い |
| 捨てやすさ | ○ 生SQLなので隔離しやすい | △ 型がアプリに染み込みやすい | △ エンティティが広がりやすい |
| 保守 | ◎ 活発 | ◎ 活発 | ◎ 活発 |

以下、特に判断を左右する観点を掘り下げます。

## AI相性・復旧性 — 一発目の精度は「罠の種類」で決まる

「AIがどれだけ正確に書けるか」はよく学習データ量の話にされます。生SQLは言語・フレームワークを超えた共通知識なので sqlx が有利、という理屈です。ただし実際に Claude Opus 4.8 に書かせて検証すると（下の実測メモ）、優劣はそう単純ではありませんでした。

- **sqlx**: 単純なCRUDではAIが膨大なSQL学習データを流用でき一発目の精度が高い。一方、集計やJOINで `query!` マクロを使うと、SQLは正しいのに**マクロ固有の作法**（`AS "col!: T"` の型・NULL注釈、その別名を `ORDER BY` で参照できない等）で一発目に躓きやすい
- **diesel**: SQLを独自DSLに翻訳する必要はあるが、AIはidiomaticに書けることが多い。外部結合の `nullable()` など**型の作法**を外しても、コンパイラが必ず弾く（誤った結果は出ない）
- **sea-orm**: ORMの発想は他言語から類推が効く。躓きどころは `PaginatorTrait` の import や join 補助メソッドの選択といった**API表面の作法**

ここで効くのが **復旧性** です。一発目を外しても、3者とも**ミスはコンパイル時に検出され、コンパイラのエラーを返すと1〜2ラウンドで全て直りました**。「AI相性（一発目）」だけを見るのではなく、「**復旧性まで含めたループの回りやすさ**」で評価するべきだということです。

:::message
**実測メモ**（Claude Opus 4.8 / 各5試行 / ドキュメント参照を禁じた「一発書き」/ スキーマは生成済みとしてクエリ本体のみ書かせ、実DBでテスト）

- 単純なブログCRUD: diesel **5/5**・sqlx **5/5**・sea-orm **2/5**
- 多対多・自己参照・LEFT JOIN集計を含む複雑モデル: diesel **3/5**・sea-orm **2/5**・sqlx **1/5**

失敗はいずれもコンパイル段階で検出され、コンパイラのエラーを返すと **1〜2ラウンドで全て修正**できた。複雑モデルでは sqlx が最も躓きやすく（4/5が `query!` マクロのエイリアス作法で失敗）、「diesel は独自DSLゆえ精度が落ちる」という単純な順位付けは観測されなかった。
:::

## ビルド — diesel はCライブラリを引きやすい

「OpenSSL/libpqが見つからない」というC依存のビルドエラーは、AIエージェントの環境構築失敗の最頻出パターンです。DBアクセスでも同じ構図が出ます。

- **sqlx**: ドライバがPure Rustで、PostgreSQL でも libpq を必要としません。TLSも rustls を選べば、`cargo build` が通る環境だけで完結します
- **diesel**: PostgreSQL バックエンドは既定で libpq（`pq-sys`）、MySQL は libmysqlclient、SQLite は libsqlite3-sys といった **C依存を引きやすい** 構成です。システムにライブラリや pkg-config が要求され、Dockerイメージやクロスコンパイル（musl静的リンク、ARM）が一段面倒になります
- **sea-orm**: 土台が sqlx なので、こちらもPure Rustで完結させられます

実際に libpq / pg_config の無い環境で PostgreSQL バックエンドをビルドすると、**diesel は `ld: library 'pq' not found` でリンクに失敗**し、sqlx / sea-orm は Pure Rust でそのまま通りました。ビルド観点では sqlx / sea-orm が有利で、diesel は環境構築ループに迷い込むリスクが相対的に高いということです。

## 残りの観点（初速・定番接続・速度・捨てやすさ・保守）

- **初速**: つなぐだけなら sea-orm が素直。sqlx は `query!` のコンパイル時検証にビルド時のDB接続か `cargo sqlx prepare` のオフラインキャッシュが要り、diesel は `diesel print-schema` でのスキーマ生成とC依存の用意が要るぶん、初回の段取りが一手間多くなります
- **定番接続**: sqlx は tokio / axum / rustls にそのまま乗り、sea-orm はその sqlx の上に構築されています。diesel は独自エコシステムの色が濃く、非同期化には `diesel-async`、接続プールには [r2d2](https://crates.io/crates/r2d2) / [deadpool](https://crates.io/crates/deadpool) など、別部品の組み合わせが要ります
- **速度**: diesel / sea-orm は型レベル計算や proc-macro が重く、ビルドが伸びがちです。`cargo check` の体感が修正ループ1回あたりの単価に直結します
- **捨てやすさ**: sqlx は生SQL文字列なので repository / adapter で隔離しやすい一方、diesel は `Queryable` / `table!` の型が、sea-orm はエンティティがアプリ側に広がりやすく、後から差し替えづらくなります
- **保守**: 3つとも活発に開発が続いており、この観点での減点はありません

## キーメッセージ: コンパイル時検証はレビューの代替になる

ここが今回の核心です。

AIが生成したコードで一番こわいのは、**それらしく書けているSQLが、実は列名を間違えている／型が合っていない** といった類のミスです。これを人間が毎回目視レビューするのは現実的ではありません。

ところが sqlx と diesel は、この種のミスを **コンパイルエラーとして弾いてくれます**。

- 存在しない列を参照した → コンパイルが通らない
- `i32` を期待する場所に `String` が来た → コンパイルが通らない
- テーブル名のタイポ → コンパイルが通らない

つまり、AIが書いたSQLの「スキーマとの整合性」という、人間レビューでもっとも見落としやすい部分を、**コンパイラがレビュアーとして肩代わり** してくれます。AIに「書く→コンパイル→直す」を回させておけば、スキーマ不整合のコードはそもそもコミットまで到達しません。

一方 **sea-orm はランタイム検証** なので、この恩恵が薄くなります。クエリの誤りはテストを書いて実行するか、本番で踏むまで分かりません。AIに任せる範囲が広いほど、「コンパイルが通った＝スキーマ整合は保証済み」という安心感の価値は大きくなります。

# 推奨: sqlx を起点にする

以上を踏まえた推奨は **sqlx** です。理由をまとめると:

1. **AI相性・復旧性**: 単純なクエリなら生SQLそのままで一発目の精度が高く、外しても具体的なエラーで直しやすい（ただし集計・JOIN等では `query!` マクロのエイリアス・NULL・型注釈の作法に一手間かかる点は割り引く）
2. **キーメッセージ**: `query!` マクロが実在スキーマと照合し、コンパイル時検証でレビューを肩代わりする
3. **定番接続・ビルド**: axum / tokio と同じ非同期エコシステムに素直に乗り、Pure Rust で `cargo build` が通る（sea-orm の土台でもある）

ただし、三つ巴である以上、他が向く場面もあります。

- **diesel が向く**: 生SQLを一切書きたくない、Rust側の型で完全に閉じたい。SQLよりRustのDSLでの一貫性を重視するチーム
- **sea-orm が向く**: 検索条件が動的に変わる管理画面やCRUD中心の業務アプリで、クエリを動的に組み立てたい。ORMの開発体験を重視する場合

「迷ったら sqlx、動的クエリ主体なら sea-orm、DSLで閉じたいなら diesel」という整理が実務的です。

# 実例: axum + sqlx の最小構成

最後に、axum と sqlx を組み合わせた最小のAPIを示します。`query!` マクロがコンパイル時にスキーマと照合される様子が見どころです。

```toml
[dependencies]
axum = "0.8"
tokio = { version = "1", features = ["full"] }
sqlx = { version = "0.8", features = ["runtime-tokio", "postgres"] }
serde = { version = "1", features = ["derive"] }
```

```rust
use axum::{extract::{Path, State}, routing::get, Json, Router};
use serde::Serialize;
use sqlx::PgPool;

#[derive(Serialize)]
struct User {
    id: i32,
    name: String,
}

async fn get_user(
    State(pool): State<PgPool>,
    Path(user_id): Path<i32>,
) -> Result<Json<User>, axum::http::StatusCode> {
    // この query! は **ビルド時に** DB スキーマと照合される。
    // 列名や型を間違えれば、ここでコンパイルが失敗する。
    let rec = sqlx::query!(
        "SELECT id, name FROM users WHERE id = $1",
        user_id
    )
    .fetch_one(&pool)
    .await
    .map_err(|_| axum::http::StatusCode::NOT_FOUND)?;

    Ok(Json(User { id: rec.id, name: rec.name }))
}

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let pool = PgPool::connect(&std::env::var("DATABASE_URL")?).await?;

    let app = Router::new()
        .route("/users/{id}", get(get_user))
        .with_state(pool);

    let listener = tokio::net::TcpListener::bind("0.0.0.0:3000").await?;
    axum::serve(listener, app).await?;
    Ok(())
}
```

ポイントは `query!` マクロです。コンパイル時に `DATABASE_URL`（または `cargo sqlx prepare` で保存したスキーマ情報）を使って `SELECT id, name FROM users` を検証するため、`users` テーブルに `name` 列が無ければ **ビルドが失敗** します。AIがこのコードを書いて `cargo build` を回した時点で、スキーマ不整合は自動的に弾かれるわけです。

:::message
`query!` のコンパイル時検証には、ビルド時にDBへ接続できるか、`cargo sqlx prepare` で生成した `.sqlx` ディレクトリ（オフライン用のスキーマキャッシュ）が必要です。CI やAIエージェントの環境では後者を使うのが定石です。
:::

# まとめ: 決着していない領域こそ、観点で判断する

前回の「決着済み」領域と違い、WebバックエンドとDBアクセスは選択肢が割れています。だからこそ、AIの提案を「なんとなく」で受け入れず、8つの観点で評価する価値があります。

今回の要点はこうです。

- Webフレームワークは、tokioエコシステムに素直に乗れる **axum** を起点に
- DBアクセスは三つ巴。**AIに書かせる前提では、生SQLで学習データ量が多くコンパイル時検証も効く sqlx** が一歩リード（ただし一発目の精度はORMの優劣より「クエリの複雑さ × crate固有の罠」で決まり、sqlx も複雑な集計・JOINでは `query!` マクロの作法に注意）
- そして **コンパイル時検証はレビューの代替になる**。AIが書いたSQLのスキーマ整合性を、コンパイラがレビュアーとして肩代わりしてくれる

次回は **時刻crate**（[chrono](https://crates.io/crates/chrono) / [time](https://crates.io/crates/time) / [jiff](https://crates.io/crates/jiff)）を扱います。「コンパイル時検証」の話が、時刻では「型でタイムゾーンの有無を分ける」という形でどう効くかを見ていきます。
