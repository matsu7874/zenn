前回（[AI時代のRust crate選定 (1) 迷わなくていい領域](#)）では、serde や tokio のように「決着済み」でAIが別の提案をしてきたら疑ってよい領域を整理しました。あわせて、AIに書かせる前提でcrateを評価するための8つの軸を定義しました。

しかしRustの全領域が決着済みなわけではありません。今回扱う **WebバックエンドAPIとデータベースアクセス** は、未だに選択肢が割れている領域です。
特にデータベースアクセスは sqlx / diesel / sea-orm の三つ巴で、しかも「AIに書かせる」前提だと、従来とは違う観点で評価が変わります。

この記事の結論を先に言うと、**コンパイル時検証はレビューの代替になる** ということです。AIが書いたSQLを人間が一行ずつ目視レビューする代わりに、コンパイラにレビュアーをやらせる。これが今回のキーメッセージです。

# crateを評価する8つの軸

まず、この連載全体で使う「ものさし」を定義する。AIの提案を評価するとき、見るべき点は次の8つだ。

1. **API安定性**: 破壊的変更の頻度と、移行ガイドの有無。注意点として、Rustでは優良crateでも0.xに留まり続ける文化がある(axumは0.8のまま広く本番利用されている)ため、「1.0未満=未成熟」という他言語の常識は使えない。
2. **採用の広がり**: crates.ioのダウンロード数と、その**推移**。絶対数より傾きを見る。
3. **エコシステム統合**: 主要crateとの連携が公式に存在するか。デファクトスタンダードなライブラリと接続できないと困る。
4. **AI学習データ量**: LLMがどれだけ正確に書けるか。新興crateはAIの一発目の精度が落ちる。
5. **Pure Rustか(C依存の有無)**: `-sys` crateはシステム側にCライブラリやpkg-configを要求し、ビルドが `cargo build` だけで完結しなくなる。代表例がTLSで、openssl-sys ではなく Pure Rust の rustls を選ぶのが標準。クロスコンパイル(musl静的リンク、ARM)、Dockerイメージの単純さ、WASM対応もこの軸で決まる
6. **依存の少なさ**: `cargo tree` で見える依存グラフ全体が、読まないままビルドに入るコードだ。依存数はサプライチェーンリスクと監査面積に直結する
7. **コンパイル時間**: proc-macroや型レベル計算の重いcrateはビルドを遅くする。`cargo build --timings` で計測でき、feature flagの絞り込み(tokioの `features = ["full"]` をやめる等)で削れることも多い。
8. **メンテナンス状況**: 最終リリース日。1年以上リリースが無いcrateは今後も更新がないと想像する。

5番目以降は、従来の選定論ではあまり語られてこなかった軸だ。しかし「AIに書かせる」前提では、いずれも意味が変わる。

- 軸5: 人間にとっての書きやすさより、AIにとっての書きやすさが生産性を決める
- 軸6: AIエージェントは「書く→コンパイル→直す」ループを1タスクで数十回回す。コンパイル時間は待ち時間ではなく、**修正ループ1回あたりの単価**として数十倍に増幅される
- 軸7: 自分がレビューできない領域ほど、読まれないままビルドに入るコードの量(=依存)を絞る意味が大きくなる
- 軸8: C依存のビルドエラー(「OpenSSLが見つからない」)はAIエージェントの環境構築失敗の最頻出パターンで、エージェントがシステムパッケージ導入のループに迷い込む原因になる。Pure Rustなら `cargo build` が通る環境さえあればよく、**環境構築ごとAIに任せられる**


# Webフレームワークは axum で考える

まずWebフレームワーク側から。前回 axum を「0.8のまま広く本番利用されている」例として挙げました。現在のRust製Webバックエンドでは、この axum がデファクトに最も近い位置にいます。

歴史的には actix-web が高いパフォーマンスで人気を集めましたが、いま新規にAIへ書かせる前提なら axum を起点に考えるのが無難です。理由は軸3（エコシステム統合）に集約されます。

- 非同期ランタイムは tokio 一択（軸: 前回参照）。axum はその tokio チームが開発しており、tokio / tower / hyper という同一エコシステムの上に乗っている
- ミドルウェアは tower の `Service` トレイトとして共通化されており、タイムアウト・リトライ・レート制限などを使い回せる
- 公式・準公式のサンプルが豊富で、軸4（AI学習データ量）の観点でもLLMが正確に書きやすい

```toml
[dependencies]
axum = "0.8"
tokio = { version = "1", features = ["full"] }
```

axum 側はこの記事の主役ではないので、ここでは「tokioエコシステムに素直に乗りたいなら axum」とだけ押さえて、本題のデータベースに進みます。

# データベースアクセスの三つ巴

Rustでデータベース（主にPostgreSQL / MySQL / SQLite）にアクセスする場合、有力な選択肢は次の3つです。設計思想がはっきり異なります。

| crate | スタイル | 検証タイミング | 一言で言うと |
|---|---|---|---|
| sqlx | 生SQL + マクロ | **コンパイル時** | SQLをそのまま書き、実在DBに照合して型を検証 |
| diesel | 型安全DSL | **コンパイル時** | SQLをRustのDSLで組み立て、型で検証 |
| sea-orm | ORM (ActiveRecord的) | ランタイム | エンティティ操作中心。動的なクエリ構築が得意 |

## sqlx: 生SQLをコンパイル時に検証する

sqlx は「SQLはSQLとして書く」というスタイルです。`query!` マクロにSQL文字列を渡すと、**ビルド時に実際のデータベース（または保存済みスキーマ情報）へ接続してクエリを検証** し、結果の列名・型からRustの構造体を生成します。

```rust
// コンパイル時に DB スキーマと照合される
let user = sqlx::query!(
    "SELECT id, name FROM users WHERE id = $1",
    user_id
)
.fetch_one(&pool)
.await?;

// user.id, user.name は型付きでアクセスできる
println!("{}: {}", user.id, user.name);
```

もしテーブル名や列名を間違えたり、型が合わなかったりすれば、**実行する前にコンパイルが失敗** します。SQL構文そのものは慣れた書き方が使え、なおかつ静的に守られるのが特徴です。

## diesel: SQLをRustのDSLで組み立てる

diesel は生SQLを書かず、Rustのコードとしてクエリを組み立てます。スキーマ定義から型を生成し、クエリビルダがすべて型レベルで整合性を保証します。

```rust
use crate::schema::users::dsl::*;

let result = users
    .filter(id.eq(user_id))
    .select((id, name))
    .first::<(i32, String)>(&mut conn)?;
```

こちらもコンパイル時に安全ですが、検証の方法が違います。sqlx が「実在DBにSQLを照合する」のに対し、diesel は「RustのDSLとして型が通るか」で守ります。SQLの知識をdiesel独自のDSLに翻訳して書く必要があり、ここがAIに書かせるうえでの分かれ目になります（後述）。

なお diesel は同期APIが基本で、非同期に乗せるには `diesel-async` を併用します。

## sea-orm: ORM として動的に組み立てる

sea-orm はいわゆるORMで、他言語のActiveRecordに近い感覚です。エンティティを定義し、それに対するメソッドでデータを操作します。非同期前提で、sqlx の上に構築されています。

```rust
let user: Option<user::Model> = User::find_by_id(user_id).one(&db).await?;
```

実行時に条件を組み立てる動的なクエリ（検索条件がリクエスト次第で変わる管理画面など）が書きやすい一方、**クエリの正しさはランタイムまで分からない** のが大きな違いです。

# 8つの軸で三つ巴を評価する

前回定義した8つの軸を、sqlx / diesel / sea-orm に当てたのが下表です。◎○△は「AIに書かせる前提での相対評価」だと思ってください。

| 軸 | sqlx | diesel | sea-orm |
|---|---|---|---|
| 1 API安定性 | ○ 0.xで時々破壊的変更 | ○ 2.xで安定（1→2は大移行だった） | ○ 1.0到達済み |
| 2 採用の広がり | ◎ 多く、伸びも強い | ○ 老舗で累計多数・横ばい気味 | △ 後発だが増加傾向 |
| 3 エコシステム統合 | ◎ tokio/axum/rustlsにそのまま乗る | △ 独自寄り（非同期は diesel-async 併用） | ○ sqlx上に構築、axum連携あり |
| 4 AI学習データ量 | ◎ 中身は標準SQL | △ 独自DSLで一発目の精度が落ちる | ○ ORMの発想は他言語から類推可 |
| 5 Pure Rust | ◎ Cドライバ不要 | △ libpq 等のC依存を引きやすい | ◎ sqlx土台でPure Rust可 |
| 6 依存の少なさ | ○ 中庸 | ○ 中庸＋システム側のCライブラリ | △ 三者で最も大きい |
| 7 コンパイル時間 | ○ マクロ分だけ増える | △ 型レベル計算で重い | △ マクロ＋sqlxで重い |
| 8 メンテナンス | ◎ 活発 | ◎ 活発 | ◎ 活発 |

以下、特に判断を左右する軸を掘り下げます。

## 軸4: AI学習データ量 — 生SQLは「みんなが知っている」

LLMがどれだけ正確に書けるかは、学習データ量に強く依存します。ここで効くのは「**SQLは言語・フレームワークを超えた共通知識**」だという点です。

- **sqlx**: クエリ本体は標準的なSQL。AIは膨大なSQLの学習データを流用でき、一発目の精度が高い
- **diesel**: diesel独自のDSLに翻訳する必要があり、AIはdiesel固有の書き方を覚えていないと書けない。バージョン間でDSLが変わると、古い学習データに引きずられた誤りも出やすい
- **sea-orm**: こちらも独自API。ただしORMの発想自体は他言語経験から類推が効く

「AIに書かせる」前提では、生SQLがそのまま使える sqlx が一歩リードします。

## 軸5: Pure Rust — diesel はCライブラリを引きやすい

前回、軸8で「OpenSSLが見つからない」というC依存のビルドエラーがAIエージェントの最頻出のつまずきだと述べました。DBアクセスでも同じ構図が出ます。

- **sqlx**: ドライバがPure Rustで、PostgreSQL でも libpq を必要としません。TLSも rustls を選べば、`cargo build` が通る環境だけで完結します
- **diesel**: PostgreSQL バックエンドは既定で libpq（`pq-sys`）、MySQL は libmysqlclient、SQLite は libsqlite3-sys といった **C依存を引きやすい** 構成です。システムにライブラリや pkg-config が要求され、Dockerイメージやクロスコンパイル（musl静的リンク、ARM）が一段面倒になります
- **sea-orm**: 土台が sqlx なので、こちらもPure Rustで完結させられます

軸8で書いた「環境構築ごとAIに任せられる」という観点では、sqlx / sea-orm が有利で、diesel は環境構築ループに迷い込むリスクが相対的に高いということです。

## 軸1: API安定性 / 軸7: コンパイル時間

- **API安定性（軸1）**: 3つとも実用十分ですが、diesel はメジャーアップデート時にDSLや導出マクロまわりの破壊的変更があり、古い学習データのAIが詰まりやすい
- **コンパイル時間（軸7）**: diesel / sea-orm は型レベル計算やproc-macroが重く、ビルドが伸びがちです。前回述べたとおり、AIエージェントは「書く→コンパイル→直す」を1タスクで何十回も回すため、コンパイル時間は待ち時間ではなく **修正ループ1回あたりの単価** として効いてきます

## 残りの軸（2・3・6・8）

決定打にはなりにくいものの、横並びで押さえておきます。

- **採用の広がり（軸2）**: 絶対数より傾きを見る軸です。diesel は老舗で累計ダウンロードは多いものの伸びは横ばい寄り、sqlx は数・伸びとも強い、sea-orm は後発で増加傾向、という位置取りです
- **エコシステム統合（軸3）**: sqlx は tokio / axum / rustls にそのまま乗り、sea-orm はその sqlx の上に構築されています。diesel は独自エコシステムの色が濃く、非同期化には `diesel-async`、接続プールには r2d2 / deadpool など、別部品の組み合わせが要ります
- **依存の少なさ（軸6）**: sqlx と diesel は中庸。sea-orm は sqlx ＋ sea-query ＋ 各種マクロを抱えるぶん、`cargo tree` が三者で最も大きくなります。読まないままビルドに入るコード量を絞りたいなら、薄い構成を選べる sqlx が有利です
- **メンテナンス状況（軸8）**: 3つとも活発に開発が続いており、この軸での減点はありません

## キーメッセージ: コンパイル時検証はレビューの代替になる

ここが今回の核心です。

AIが生成したコードで一番こわいのは、**それらしく書けているSQLが、実は列名を間違えている／型が合っていない** といった類のミスです。これを人間が毎回目視レビューするのは現実的ではありません。

ところが sqlx と diesel は、この種のミスを **コンパイルエラーとして弾いてくれます**。

- 存在しない列を参照した → コンパイルが通らない
- `i32` を期待する場所に `String` が来た → コンパイルが通らない
- テーブル名のタイポ → コンパイルが通らない

つまり、AIが書いたSQLの「スキーマとの整合性」という、人間レビューでもっとも見落としやすい部分を、**コンパイラがレビュアーとして肩代わり** してくれます。AIに「書く→コンパイル→直す」を回させておけば、スキーマ不整合のコードはそもそもコミットまで到達しません。

一方 **sea-orm はランタイム検証** なので、この恩恵が薄くなります。クエリの誤りはテストを書いて実行するか、本番で踏むまで分かりません。AIに任せる範囲が広いほど、「コンパイルが通った＝スキーマ整合は保証済み」という安心感の価値は大きくなります。

# 推奨: sqlx を起点にする

以上を踏まえた推奨は **sqlx** です。理由をまとめると:

1. **軸4**: クエリが生SQLなので、AIの一発目の精度が高い
2. **キーメッセージ**: `query!` マクロが実在スキーマと照合し、コンパイル時検証でレビューを肩代わりする
3. **軸3**: axum / tokio と同じ非同期エコシステムに素直に乗る（sea-orm の土台でもある）

ただし、三つ巴である以上、他が向く場面もあります。

- **diesel が向く**: 生SQLを一切書きたくない、Rust側の型で完全に閉じたい。SQLよりRustのDSLでの一貫性を重視するチーム
- **sea-orm が向く**: 検索条件が動的に変わる管理画面やCRUD中心の業務アプリで、クエリを動的に組み立てたい。ORMの開発体験を重視する場合

「迷ったら sqlx、動的クエリ主体なら sea-orm、DSLで閉じたいなら diesel」という整理が実務的です。

# 実例: axum + sqlx の最小構成

最後に、axum と sqlx を組み合わせた最小のAPIを示します。`query!` マクロがコンパイル時にスキーマと照合される様子が見どころです。

```toml
[dependencies]
axum = "0.8"
tokio = { version = "1", features = ["full"] }
sqlx = { version = "0.8", features = ["runtime-tokio", "postgres"] }
serde = { version = "1", features = ["derive"] }
```

```rust
use axum::{extract::{Path, State}, routing::get, Json, Router};
use serde::Serialize;
use sqlx::PgPool;

#[derive(Serialize)]
struct User {
    id: i32,
    name: String,
}

async fn get_user(
    State(pool): State<PgPool>,
    Path(user_id): Path<i32>,
) -> Result<Json<User>, axum::http::StatusCode> {
    // この query! は **ビルド時に** DB スキーマと照合される。
    // 列名や型を間違えれば、ここでコンパイルが失敗する。
    let rec = sqlx::query!(
        "SELECT id, name FROM users WHERE id = $1",
        user_id
    )
    .fetch_one(&pool)
    .await
    .map_err(|_| axum::http::StatusCode::NOT_FOUND)?;

    Ok(Json(User { id: rec.id, name: rec.name }))
}

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let pool = PgPool::connect(&std::env::var("DATABASE_URL")?).await?;

    let app = Router::new()
        .route("/users/{id}", get(get_user))
        .with_state(pool);

    let listener = tokio::net::TcpListener::bind("0.0.0.0:3000").await?;
    axum::serve(listener, app).await?;
    Ok(())
}
```

ポイントは `query!` マクロです。コンパイル時に `DATABASE_URL`（または `cargo sqlx prepare` で保存したスキーマ情報）を使って `SELECT id, name FROM users` を検証するため、`users` テーブルに `name` 列が無ければ **ビルドが失敗** します。AIがこのコードを書いて `cargo build` を回した時点で、スキーマ不整合は自動的に弾かれるわけです。

:::message
`query!` のコンパイル時検証には、ビルド時にDBへ接続できるか、`cargo sqlx prepare` で生成した `.sqlx` ディレクトリ（オフライン用のスキーマキャッシュ）が必要です。CI やAIエージェントの環境では後者を使うのが定石です。
:::

# まとめ: 決着していない領域こそ、軸で判断する

前回の「決着済み」領域と違い、WebバックエンドとDBアクセスは選択肢が割れています。だからこそ、AIの提案を「なんとなく」で受け入れず、8つの軸で評価する価値があります。

今回の要点はこうです。

- Webフレームワークは、tokioエコシステムに素直に乗れる **axum** を起点に
- DBアクセスは三つ巴。**AIに書かせる前提では、生SQLで学習データ量が多くコンパイル時検証も効く sqlx** が一歩リード
- そして **コンパイル時検証はレビューの代替になる**。AIが書いたSQLのスキーマ整合性を、コンパイラがレビュアーとして肩代わりしてくれる
