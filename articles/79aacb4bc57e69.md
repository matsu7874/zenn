前回（[AI時代のRust crate選定 (1) デファクトスタンダードなcrate群](https://zenn.dev/estie/articles/4e057582208b0b)）の記事では、[serde](https://crates.io/crates/serde) や [tokio](https://crates.io/crates/tokio) のようにデファクトスタンダードなcrateがある領域を紹介しました。この領域でAIが別の提案をしてきたら特別の意図を確認するべきと整理しました。

Rustの全ての領域でデファクトスタンダードなライブラリがあるわけではありません。今回扱うWebバックエンドとデータベースアクセスは、有力な少数の選択肢から選ぶ必要のある領域です。
Webバックエンド側は axum を起点に考える前提だけ確認し、記事の主眼はデータベースアクセスcrateの比較に置きます。
特にデータベースアクセスは [sqlx](https://crates.io/crates/sqlx) / [diesel](https://crates.io/crates/diesel) / [sea-orm](https://crates.io/crates/sea-orm) と有力な選択肢が３つあります。

この記事の結論を先に言うと、**コンパイル時検証はスキーマ整合性レビューの一部を肩代わりする** ということです。AIが書いたSQLを人間の目視レビューだけに頼らず、列名・型・テーブル名の不整合をコンパイラに弾かせます。これが今回のキーメッセージです。

# crateを評価する4つの観点

機能が足りているかはAIに判断させる前提で、人間が見るべきは**AIに書かせたときに実装ループが回るか**です。観点を「チェック内容」と「NG例」で4つに整理しました。

| 観点 | チェック内容 | NG例 |
|---|---|---|
| AI親和性 | AIが現行APIでコードを書け、エラーを渡せば修正できる | 古い/存在しないAPIを出し、エラーを見ても修正できない |
| 立ち上げと反復速度 | READMEから環境構築が簡単で、build と `cargo check` の待ちが短い | feature指定やC依存で詰まり開発を始められない、修正に時間がかかる |
| エコシステム適合 | tokio/serde/tracing等と自然に繋がる | 独自世界観が強すぎる |
| 撤退可能性 | adapterで隔離でき、メンテが止まっても捨てられる | アプリ全体に染み込み、放置されても逃げられない |

どれも「人間にとっての書きやすさ」ではなく、AIに書かせたときに実装ループが回るかを見ているのがポイントです。4観点の背景を補足します。

- **AI親和性**: AIエージェントは「書く→コンパイル→修正」ループを1タスクで何十回も回します。一発目が多少外れても、エラーが具体的で修正できるなら実用上の損失は小さい。逆に、エラーが難解で修正できないとエージェントが詰まってしまいます。普段馴染みのない領域を扱うときほど、AIが使えるライブラリを選ぶ価値が大きくなります。
- **立ち上げと反復速度**: READMEからシンプルに動くか、`cargo build` / `cargo check` の待ちが短いかは「編集→確認」ループごとのコストです。特にC依存のビルドエラー(「OpenSSL/libpqが見つからない」)はAIエージェントの環境構築失敗の最頻出パターンです。Pure Rustで完結する構成なら `cargo build` が通る環境さえあればよく、環境構築ごとAIに任せられます。コンパイル時間も、修正ループ1回あたりのコストとして数十回ぶん積み上がります。
- **エコシステム適合**: 周辺crateと自然に繋がるかを見ます。Webバックエンドなら tokio / tower / serde / tracing / rustls などと素直に組み合わせられるほど、AIが既存のサンプルやエラーから修正しやすくなります。逆に独自の抽象やランタイムに寄るほど、周辺知識の翻訳コストが増えます。
- **撤退可能性**: AIに任せると依存はどんどん増えます。それでも怖くないのは、adapterで隔離して、ダメになったら捨てられるからです。馴染みのない領域のvibe codingでは、crateを育てる側ではなく利用者側に回ることが多いはずです。だからこそ「保守が完璧か」より「捨てやすいか」を重視します。メンテが細っても、隔離してあれば差し替えられます。


# Webフレームワークは axum で考える

まずWebフレームワーク側から見ます。現在のRust製Webバックエンドでは、tokioエコシステムに素直に乗るという観点で [axum](https://crates.io/crates/axum) が最も無難な起点です。

歴史的には [actix-web](https://crates.io/crates/actix-web) が高いパフォーマンスで人気を集めました。actix-web が不適切という話ではありませんが、いま新規にAIへ書かせる前提なら axum を起点に考えるのが無難です。理由はエコシステム適合（tokioエコシステムにそのまま乗れるか）に集約されます。

- 非同期ランタイムは tokio 一強（観点: 前回参照）。axum はその tokio チームが開発しており、tokio / [tower](https://crates.io/crates/tower) / [hyper](https://crates.io/crates/hyper) という同一エコシステムの上に乗っている
- ミドルウェアは tower の `Service` トレイトとして共通化されており、タイムアウトやリトライ、レート制限などを使い回せる
- 公式や準公式のサンプルが豊富で、AI親和性の観点でもLLMが正確に書きやすい

```toml
[dependencies]
axum = "0.8"
tokio = { version = "1", features = ["full"] }
```

axum 側はこの記事の主役ではないので、ここでは「tokioエコシステムにシンプルに乗りたいなら axum」とだけ押さえて、本題のデータベースに進みます。

# データベースアクセスの三つ巴

Rustでデータベース（主にPostgreSQL / MySQL / SQLite）にアクセスする場合、有力な選択肢は次の3つです。それぞれ設計思想が大きく異なります。

| crate | スタイル | 検証タイミング | 一言で言うと |
|---|---|---|---|
| sqlx | 生SQL + マクロ | **コンパイル時** | SQLをそのまま書き、実在DBに照合して型を検証 |
| diesel | 型安全DSL | **コンパイル時** | SQLをRustのDSLで組み立て、型で検証 |
| sea-orm | ORM (ActiveRecord的) | Rust側の型 + ランタイム | エンティティ操作中心。動的なクエリ構築が得意 |

## sqlx: 生SQLをコンパイル時に検証する

sqlx は「SQLはSQLとして書く」というスタイルです。`query!` マクロにSQL文字列を渡すと、ビルド時に実際のデータベース（または保存済みスキーマ情報）へ接続してクエリを検証し、結果の列名や型からRustの構造体を生成します。

```rust
// コンパイル時に DB スキーマと照合される
let user = sqlx::query!(
    "SELECT id, name FROM users WHERE id = $1",
    user_id
)
.fetch_one(&pool)
.await?;

// user.id, user.name は型付きでアクセスできる
println!("{}: {}", user.id, user.name);
```

もしテーブル名や列名を間違えたり、型が合わなかったりすれば、実行する前にコンパイルが失敗します。SQL構文そのものは慣れた書き方が使え、なおかつ静的に守られるのが特徴です。

## diesel: SQLをRustのDSLで組み立てる

diesel は生SQLを書かず、Rustのコードとしてクエリを組み立てます。スキーマ定義から型を生成し、クエリビルダがすべて型レベルで整合性を保証します。

```rust
use crate::schema::users::dsl::*;

let result = users
    .filter(id.eq(user_id))
    .select((id, name))
    .first::<(i32, String)>(&mut conn)?;
```

こちらもコンパイル時に安全ですが、検証の方法が違います。sqlx が「実在DBにSQLを照合する」のに対し、diesel は「RustのDSLとして型が通るか」で守ります。SQLの知識をdiesel独自のDSLに翻訳して書く必要があり、ここがAIに書かせるうえでの分かれ目になります（後述）。

なお diesel は同期APIが基本で、非同期に乗せるには [`diesel-async`](https://crates.io/crates/diesel-async) を併用します。

## sea-orm: ORM として動的に組み立てる

sea-orm はいわゆるORMで、他言語のActiveRecordに近い感覚です。エンティティを定義し、それに対するメソッドでデータを操作します。非同期前提で、sqlx の上に構築されています。

```rust
let user: Option<user::Model> = User::find_by_id(user_id).one(&db).await?;
```

実行時に条件を組み立てる動的なクエリ（検索条件がリクエスト次第で変わる管理画面など）が書きやすい一方、sqlx の `query!` のように **実在DBへ照合してSQL文字列をコンパイル時検証する** 仕組みとは性質が違います。生成されたエンティティやカラム名の型で守られる範囲はありますが、最終的なクエリの意味やDBスキーマとのずれは、テスト実行まで残りやすくなります。

# 4つの観点で三つ巴を評価する

先ほどのチェックリストを sqlx / diesel / sea-orm に当てたのが下表です。◎○△は「AIに書かせる前提での相対評価」だと思ってください。

| 観点 | sqlx | diesel | sea-orm |
|---|---|---|---|
| AI親和性 | ◎ 単純SQLは一発、外しても具体的エラーで修正できる（集計/JOINはマクロ作法に注意） | ○ idiomaticに書け、外しても型が必ず弾いて修正できる | ○ API作法で躓くが、import不足等は修正しやすい |
| 立ち上げと反復速度 | ○ PostgreSQL / MySQL はPure Rust。`query!`検証にDB接続/オフラインキャッシュが要る | △ libpq等のC依存＋schema生成。型レベル計算でビルドも重い | ○ 繋ぐだけはシンプル。PostgreSQL / MySQL はPure Rust（マクロ＋sqlxでビルドはやや重い） |
| エコシステム適合 | ◎ tokio/axum/[rustls](https://crates.io/crates/rustls)にそのまま | △ 独自寄り（非同期は diesel-async） | ○ sqlx上に構築 |
| 撤退可能性 | ◎ 生SQLで隔離しやすく、開発も活発 | △ 型がアプリに染み込み差し替えづらい（開発は活発） | △ エンティティが広がりやすい（開発は活発） |

以下、特に判断を左右する観点を順に見ます。

## AI親和性は一発目の精度より修正のしやすさで決まる

「AIがどれだけ正確に書けるか」はよく学習データ量の話にされます。生SQLは言語やフレームワークを超えた共通知識なので sqlx が有利、という理屈です。ただし実際に Claude Opus 4.8 に書かせて検証すると（下の実測メモ）、優劣はそう単純ではありませんでした。

- **sqlx**: 単純なCRUDではAIが膨大なSQL学習データを流用でき一発目の精度が高い。一方、集計やJOINで `query!` マクロを使うと、SQLは正しいのにマクロ固有の作法（`AS "col!: T"` の型やNULL注釈、その別名を `ORDER BY` で参照できない等）で一発目に躓きやすい
- **diesel**: SQLを独自DSLに翻訳する必要はあるが、AIはidiomaticに書けることが多い。外部結合の `nullable()` など型の作法を外しても、コンパイラが必ず弾く（誤った結果は出ない）
- **sea-orm**: ORMの発想は他言語から類推が効く。躓きどころは `PaginatorTrait` の import や join 補助メソッドの選択といったAPI表面の作法

ここで効くのがAI親和性の後半、「修正できるか」です。一発目を外しても、今回の失敗はコンパイルエラーまたは実DBテストの失敗として検出でき、エラーを返すと1〜2ラウンドで全て修正できました。AI親和性は一発目の精度だけでなく、修正できるか（復旧まで含めたループの回りやすさ）で評価するべきだということです。

:::message
**実測メモ**（Claude Opus 4.8 / 各5試行 / ドキュメント参照を禁じた「一発書き」/ スキーマは生成済みとしてクエリ本体のみ書かせ、実DBでテスト）

これは著者環境での小さな検証メモで、汎用ベンチマークではありません。ここで見たいのは勝率そのものより、「外したときにエラーから復旧できるか」です。

- 単純なブログCRUD: diesel **5/5**・sqlx **5/5**・sea-orm **2/5**
- 多対多・自己参照・LEFT JOIN集計を含む複雑モデル: diesel **3/5**・sea-orm **2/5**・sqlx **1/5**

失敗はいずれもコンパイルまたは実DBテストの段階で検出され、エラーを返すと **1〜2ラウンドで全て修正**できた。複雑モデルでは sqlx が最も躓きやすく（4/5が `query!` マクロのエイリアス作法で失敗）、「diesel は独自DSLゆえ精度が落ちる」という単純な順位付けは観測されなかった。
:::

## 立ち上げと反復速度では diesel がCライブラリを引きやすい

「OpenSSL/libpqが見つからない」というC依存のビルドエラーは、AIエージェントの環境構築失敗の最頻出パターンです。DBアクセスでも同じ構図が出ます。

- **sqlx**: PostgreSQL / MySQL ドライバはPure Rustで、PostgreSQL でも libpq を必要としません。TLSも rustls を選べば、`cargo build` が通る環境だけで完結します。SQLite は `libsqlite3-sys` を使うため、ここでは主にPostgreSQL / MySQL接続の話として読んでください
- **diesel**: PostgreSQL バックエンドは既定で libpq（`pq-sys`）、MySQL は libmysqlclient、SQLite は libsqlite3-sys といったC依存を引きやすい構成です。システムにライブラリや pkg-config が要求され、Dockerイメージやクロスコンパイル（musl静的リンク、ARM）が一段面倒になります
- **sea-orm**: 土台が sqlx なので、PostgreSQL / MySQL ではこちらもPure Rustで完結させられます

実際に libpq / pg_config の無い環境で PostgreSQL バックエンドをビルドすると、diesel は `ld: library 'pq' not found` でリンクに失敗し、sqlx / sea-orm はそのまま通りました。立ち上げの観点では sqlx / sea-orm が有利で、diesel は環境構築ループに迷い込むリスクが相対的に高いということです。一方で、diesel の型安全DSLそのものは成熟しており、環境を整えられるチームでは強い選択肢です。

初回の段取り（初速）でも差が出ます。つなぐだけなら sea-orm がシンプル。sqlx は `query!` のコンパイル時検証にビルド時のDB接続か `cargo sqlx prepare` のオフラインキャッシュが要り、diesel は `diesel print-schema` でのスキーマ生成とC依存の用意が要るぶん、初回の手数が多くなります。反復速度（`cargo check`）も diesel / sea-orm は型レベル計算や proc-macro が重く伸びがちで、修正ループ1回あたりの単価に効いてきます。

## 残りの観点（エコシステム適合と撤退可能性）

- **エコシステム適合**: sqlx は tokio / axum / rustls にそのまま乗り、sea-orm はその sqlx の上に構築されています。diesel は独自エコシステムの色が濃く、非同期化には `diesel-async`、接続プールには [r2d2](https://crates.io/crates/r2d2) / [deadpool](https://crates.io/crates/deadpool) など、別部品の組み合わせが要ります
- **撤退可能性**: sqlx は生SQL文字列なので repository / adapter で隔離しやすく、ダメになっても差し替えやすい。一方 diesel は `Queryable` / `table!` の型が、sea-orm はエンティティがアプリ側に広がりやすく、後から剥がしづらくなります。3つとも開発は活発で「いま保守が止まる」心配は小さいですが、利用者側として保守コストを負えない場合は「いざという時に捨てられるか」を優先します

## コンパイル時検証はレビューの一部を肩代わりする

AIが生成したコードで一番こわいのは、それらしく書けているSQLが実は列名を間違えている、型が合っていない、といった類のミスです。これを人間が毎回目視レビューだけで拾うのは現実的ではありません。

ところが sqlx と diesel は、この種のミスをコンパイルエラーとして弾いてくれます。

- 存在しない列を参照した → コンパイルが通らない
- `i32` を期待する場所に `String` が来た → コンパイルが通らない
- テーブル名のタイポ → コンパイルが通らない

つまり、AIが書いたSQLの「スキーマとの整合性」という、人間レビューでもっとも見落としやすい部分を、**コンパイラがレビュアーとして一部肩代わり**してくれます。AIに「書く→コンパイル→修正」を回させておけば、少なくとも列名・型・テーブル名の不整合はコミット前に弾けます。

もちろん、コンパイル時検証はレビュー全体の代替ではありません。WHERE条件の意味、JOINの向き、N+1、権限条件の抜け、インデックス設計、NULL推論の限界などは別途レビューやテストが必要です。一方 sea-orm は、生成された型で守られる部分はあるものの、実在DBへSQLを照合する `query!` 型の恩恵は薄くなります。AIに任せる範囲が広いほど、「コンパイルが通った＝少なくともスキーマ整合の大部分は確認済み」という安心感の価値は大きくなります。

# 推奨は sqlx を起点にすること

以上を踏まえた推奨は sqlx です。理由をまとめると次のとおりです。

1. **AI親和性**: 単純なクエリなら生SQLそのままで一発目の精度が高く、外しても具体的なエラーで修正しやすい（ただし集計やJOIN等では `query!` マクロのエイリアス、NULL、型注釈の作法に一手間かかる点は割り引く）
2. **キーメッセージ**: `query!` マクロが実在スキーマと照合し、コンパイル時検証でスキーマ整合性レビューの一部を肩代わりする
3. **エコシステム適合と立ち上げ**: axum / tokio と同じ非同期エコシステムにシンプルに乗り、PostgreSQL / MySQL ではPure Rustで `cargo build` が通る（sea-orm の土台でもある）

ここでの推奨は、一発書きの勝率だけで決めているわけではありません。複雑なクエリの一発書きでは diesel に負けるケースがあっても、初速・環境構築・撤退可能性まで含めた実務の既定値として sqlx を起点に置く、という重み付けです。

ただし、三つ巴である以上、他が向く場面もあります。

- **diesel が向く**: 生SQLを一切書きたくない、Rust側の型で完全に閉じたい。SQLよりRustのDSLでの一貫性を重視するチーム
- **sea-orm が向く**: 検索条件が動的に変わる管理画面やCRUD中心の業務アプリで、クエリを動的に組み立てたい。ORMの開発体験を重視する場合

「迷ったら sqlx、動的クエリ主体なら sea-orm、DSLで閉じたいなら diesel」という整理が実務的です。

# axum + sqlx の最小構成

最後に、axum と sqlx を組み合わせた最小のAPIを示します。`query!` マクロがコンパイル時にスキーマと照合される様子が見どころです。

```toml
[dependencies]
axum = "0.8"
tokio = { version = "1", features = ["full"] }
sqlx = { version = "0.8", features = ["runtime-tokio", "postgres"] }
serde = { version = "1", features = ["derive"] }
anyhow = "1"
```

```rust
use axum::{extract::{Path, State}, routing::get, Json, Router};
use serde::Serialize;
use sqlx::PgPool;

#[derive(Serialize)]
struct User {
    id: i32,
    name: String,
}

async fn get_user(
    State(pool): State<PgPool>,
    Path(user_id): Path<i32>,
) -> Result<Json<User>, axum::http::StatusCode> {
    // この query! は **ビルド時に** DB スキーマと照合される。
    // 列名や型を間違えれば、ここでコンパイルが失敗する。
    let rec = sqlx::query!(
        "SELECT id, name FROM users WHERE id = $1",
        user_id
    )
    .fetch_one(&pool)
    .await
    .map_err(|_| axum::http::StatusCode::NOT_FOUND)?;

    Ok(Json(User { id: rec.id, name: rec.name }))
}

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let pool = PgPool::connect(&std::env::var("DATABASE_URL")?).await?;

    let app = Router::new()
        .route("/users/{id}", get(get_user))
        .with_state(pool);

    let listener = tokio::net::TcpListener::bind("0.0.0.0:3000").await?;
    axum::serve(listener, app).await?;
    Ok(())
}
```

ポイントは `query!` マクロです。コンパイル時に `DATABASE_URL`（または `cargo sqlx prepare` で保存したスキーマ情報）を使って `SELECT id, name FROM users` を検証するため、`users` テーブルに `name` 列が無ければビルドが失敗します。AIがこのコードを書いて `cargo build` を回した時点で、スキーマ不整合は自動的に弾かれます。

:::message
`query!` のコンパイル時検証には、ビルド時にDBへ接続できるか、`cargo sqlx prepare` で生成した `.sqlx` ディレクトリ（オフライン用のスキーマキャッシュ）が必要です。CI やAIエージェントの環境では後者を使うのが定石です。
:::

# 決着していない領域は観点で判断する

前回の「決着済み」領域と違い、WebバックエンドとDBアクセスは選択肢が割れています。だからこそ、AIの提案を「なんとなく」で受け入れず、4つの観点で評価する価値があります。

今回の要点は次の3つです。

- Webフレームワークは、tokioエコシステムにシンプルに乗れる axum を起点にする
- DBアクセスは三つ巴で、AIに書かせる前提では、生SQLで学習データ量が多くコンパイル時検証も効く sqlx が一歩リードする（ただし一発目の精度はORMの優劣より「クエリの複雑さ × crate固有の罠」で決まり、sqlx も複雑な集計やJOINでは `query!` マクロの作法に注意がいる）
- コンパイル時検証はレビューの一部を肩代わりする。AIが書いたSQLのスキーマ整合性を、コンパイラがコミット前に弾いてくれる

次回は時刻crate（[chrono](https://crates.io/crates/chrono) / [time](https://crates.io/crates/time) / [jiff](https://crates.io/crates/jiff)）を扱います。「コンパイル時検証」の話が、時刻では「型でタイムゾーンの有無を分ける」という形でどう効くかを見ていきます。
