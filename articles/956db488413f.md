[やまびこスピーチ](https://yamabiko.matsu7874.work/) というWebアプリを作りました。ランダムなお題について話すと、ラウンド2以降に、過去の**やまびこ**があなた自身の声で聞こえてくる「フレーズで遊ぶ即興ループ」です。

![やまびこスピーチのタイトル画面](/images/yamabiko-speech-title.png)
*公開版のタイトル画面。AmiVoiceとClaudeを使う外部API構成で動かしています。*

アプリ全体の説明は省きます。この記事では、AmiVoice APIをどう使い、認識結果をどうLLMに渡す形へ設計したかに絞ります。

# 何をしたかったか

やりたかった体験は、単なる文字起こしではありません。

1. ユーザーがしゃべる
2. 音声認識で「何を言ったか」と「いつ言ったか」を取る
3. LLMが、その中から印象的な短いフレーズを選ぶ
4. 選ばれた範囲の音声を、あなた自身の声のまま再生する

つまり必要なのは、全文テキストだけではなく、**単語ごとの開始時刻・終了時刻**です。

全文だけなら、LLMに「フレーズを選んで」と頼むだけで済みます。しかし、そのフレーズを「やまびこ」としてあなた自身の声で聞こえるようにするには、元音声の何秒から何秒を再生すればよいかが必要です。ここでAmiVoice APIの `tokens` が効きました。

# AmiVoice APIの同期HTTPインタフェースを使う

[AmiVoice APIのクイックスタート](https://docs.amivoice.com/amivoice-api/manual/getting-started/)では、同期HTTPインタフェースを次のように呼び出します。

```bash
curl https://acp-api.amivoice.com/v1/recognize \
  -F d=-a-general \
  -F u=$API_KEY \
  -F a=@test.wav
```

`やまびこスピーチ` でもこの形を使っています。実装では、ブラウザ側で録音した音声を16kHz mono WAVにして、バックエンドからAmiVoice APIへ送ります。

```python
resp = httpx.post(
    "https://acp-api.amivoice.com/v1/recognize",
    data={
        "u": appkey,
        "d": "-a-general",
    },
    files={
        "a": ("audio.wav", wav_bytes, "audio/wav"),
    },
    timeout=60,
)
payload = resp.json()
```

`u` はAPPKEY、`d` は接続エンジンです。今回は日本語の即興スピーチが対象なので、汎用日本語の `-a-general` を既定にしました。

# 欲しかったのは `text` ではなく `tokens`

[音声認識の結果フォーマット](https://docs.amivoice.com/amivoice-api/manual/result-format)を見ると、AmiVoice APIのレスポンスには、全体の認識結果 `text` に加えて、単語単位の情報が `results[].tokens[]` に入ります。

```json
{
  "results": [
    {
      "tokens": [
        {
          "written": "アドバンスト・メディア",
          "confidence": 1,
          "starttime": 522,
          "endtime": 1578,
          "spoken": "あどばんすとめでぃあ"
        },
        {
          "written": "は",
          "confidence": 1,
          "starttime": 1578,
          "endtime": 1866,
          "spoken": "は"
        }
      ],
      "text": "アドバンスト・メディアは..."
    }
  ],
  "text": "アドバンスト・メディアは...",
  "message": ""
}
```

今回使った主なフィールドは次の3つです。

| フィールド | 用途 |
|---|---|
| `written` | LLMに渡す単語表記、画面に出すテキスト |
| `starttime` | その単語の開始時刻 |
| `endtime` | その単語の終了時刻 |

`starttime` / `endtime` はミリ秒なので、アプリ内では秒へ変換します。

```python
words = []
for seg in payload.get("results", []):
    for tok in seg.get("tokens", []):
        text = (tok.get("written") or "").strip()
        if not text or text == "_":
            continue

        words.append({
            "i": len(words),
            "t": float(tok.get("starttime", 0)) / 1000.0,
            "end": float(tok.get("endtime", 0)) / 1000.0,
            "text": text,
        })
```

ここで `i` という通し番号を振っているのが重要です。以降は、LLMにもアプリにも、この番号付き単語列を共通形式として渡します。

# LLMにはフレーズ文字列ではなく「単語範囲」を選ばせる

最初は、LLMにフレーズ文字列だけを出力させるつもりでした。しかし、この形だと困ります。LLMが出力した文字列を元の認識結果から再検索して、音声の開始秒・終了秒を推定しなければならないからです。

日本語の音声認識では、句読点、助詞、表記ゆれ、フィラーの有無で完全一致が壊れます。ここに曖昧検索を入れるより、LLMには最初から「どの単語からどの単語までか」を選ばせる方が安定します。

そのため、LLMには単語列を番号付きで渡し、`startIndex` と `endIndex` で範囲を出力させます。実際に使っている system prompt はこれです。

```text
あなたは日本語の話し言葉から「気持ちいいフレーズ」を見つける編集者です。語感が強く、意外性・断定・比喩・リズムがあり、字幕で大きく出すと映える短いフレーズ(おおむね10〜20文字)を選びます。出力は指定された JSON のみ。説明文やコードフェンスは出力しないこと。
```

user prompt のテンプレートは次の形です。`WORD_COUNT` 以降に、AmiVoice APIの `tokens` から作った単語列を入れています。

```text
お題: {topic or "(指定なし)"}
以下は1分間の発話を単語単位で番号付けしたものです。各行は [通し番号|開始秒] 単語。

WORD_COUNT={len(words)}
[{i}|{t}] {text}
...

この中から「気持ちいいフレーズ」をちょうど {n} 個選んでください。
各フレーズは連続する単語の範囲(startIndex〜endIndex)で指定します。
範囲は短く(目安1〜4秒、20文字以内)。displayText は字幕表示用に句読点を整えてよいが、元の語順・意味は変えないこと。

次の JSON だけを出力:
{"phrases":[{"startIndex":int,"endIndex":int,"displayText":string,"reason":string}]}
```

期待する出力形式はこうです。

```json
{
  "phrases": [
    {
      "startIndex": 1,
      "endIndex": 5,
      "displayText": "カレーは記憶の圧力",
      "reason": "比喩が強く、字幕として印象に残る"
    }
  ]
}
```

この設計にすると、時刻の決定はLLMではなくアプリ側でできます。`startIndex` の単語の `t` を開始秒にし、`endIndex` の単語の `end` を終了秒にすればよいだけです。

LLMは「気持ちいいフレーズ」を選び、AmiVoiceのタイムスタンプは「その声がどこにあるか」を決める。役割を分けると、実装がかなり単純になります。

# チャンク認識では時刻を足し戻す

このアプリでは、発話が終わってからまとめて認識するのではなく、録音中に短いチャンクを順次AmiVoice APIへ送っています。

理由は、待ち時間を減らすためです。ラウンド終了後に音声全体を送ると、その間ユーザーが待つことになります。録音中から先回りして認識しておけば、ラウンド終了時には末尾だけを処理すれば済みます。

ここで注意が必要なのは、AmiVoice APIの `starttime` / `endtime` は、送信した音声ファイルの先頭からの時刻だということです。14秒目から28秒目までのチャンクを送った場合、レスポンス内の `starttime=1000` は「ラウンド開始から15秒」ではなく、「そのチャンクの先頭から1秒」です。

そのため、チャンクの開始秒を `t` と `end` に足して、ラウンド全体の時刻に戻します。この補正をしておけば、LLMへ渡す単語列は、チャンク分割を意識しない1本のタイムラインになります。

# エラーと無音は「単語が取れなかった」に寄せる

音声アプリでは、ユーザーが何も話さない、短すぎる、マイクが拾えていない、といったケースが普通に起きます。

AmiVoice APIのレスポンスでは、正常時は `message` が空で、失敗時は `message` に理由が入ります。実装では、APPKEY不備など明確な失敗はエラーにしつつ、信頼度不足で棄却された発話は「単語が0件」として扱いました。

ゲーム側では、1ラウンドで単語が0件なら「無音だった」と判断して終了します。認識できなかった音声を無理にLLMへ渡すより、空の単語列として早めに扱う方が、後段の設計が単純でした。

# この設計でよかったこと

今回の設計を一言でまとめると、AmiVoice APIの結果を `{ i, t, end, text }` という中間表現に変換したことです。そこからLLMに `{ startIndex, endIndex, displayText }` を出力させ、最後に音声の `start/end` 区間をあなた自身の声のやまびことして再生します。

この形にしたことで、次の点が扱いやすくなりました。

- LLMに秒数を推測させなくてよい
- LLMの表記ゆれで音声区間を見失わない
- STTバックエンドを差し替えても、同じ中間表現にすれば後段が変わらない
- 「フレーズとして使える」と「音声として再生できる」が同じデータでつながる

特に最後が重要でした。生成AIを使うと、つい新しい文章を生成したくなります。しかし今回の体験では、LLMは文章生成器というより編集者です。AmiVoice APIが作った時間付き単語列から、どの範囲をやまびことして鳴らすかを選ぶ役割にしました。

# まとめ

`やまびこスピーチ` では、AmiVoice APIを「音声をテキストに変換するAPI」としてだけでなく、**あなた自身の声でやまびこを作るための時間付き単語列を作るAPI**として使いました。

今回の学びは次の3つです。

- 音声体験では、全文の `text` より `tokens[].starttime/endtime` が効く
- LLMにはフレーズ文字列ではなく、単語の `startIndex/endIndex` を出力させると安定する
- チャンク認識する場合は、チャンク開始秒を足し戻して1本のタイムラインにする

会議録や要約だけでなく、音声そのものをUIに戻す用途でも、AmiVoice APIとLLMの組み合わせは使いやすいです。テキストと時刻が一緒に取れると、「読ませるための文字起こし」から一歩進んで、「音声を素材として再構成する」設計ができます。
